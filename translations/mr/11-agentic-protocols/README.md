# एजेंटिक प्रोटोकॉल्सचा वापर (MCP, A2A आणि NLWeb)

[![Agentic Protocols](../../../translated_images/mr/lesson-11-thumbnail.b6c742949cf1ce2a.webp)](https://youtu.be/X-Dh9R3Opn8)

> _(या धड्याचा व्हिडिओ पाहण्यासाठी वरील प्रतिमेवर क्लिक करा)_

AI एजंट्सचा वापर वाढत असल्याने, मानकीकरण, सुरक्षा यासाठी आणि खुल्या नवोपक्रमाला समर्थन देणाऱ्या प्रोटोकॉल्सची गरज देखील वाढत आहे. या धड्यात आपण या गरजेला पूर्ण करणार्‍या 3 प्रोटोकॉल्सवर चर्चा करू - मॉडेल कॉन्टेक्स्ट प्रोटोकॉल (MCP), एजंट टू एजंट (A2A) आणि नॅचरल लँग्वेज वेब (NLWeb).

## परिचय

या धड्यात आपण खालील गोष्टी पाहणार आहोत:

• **MCP** कसे AI एजंट्सना बाह्य साधने आणि डेटा वापरून वापरकर्त्याचे कार्य पूर्ण करण्याची परवानगी देते.

• **A2A** कसे विविध AI एजंट्स दरम्यान संवाद आणि सहकार्य सुलभ करते.

• **NLWeb** कसे कोणत्याही संकेतस्थळावर नैसर्गिक भाषा इंटरफेस आणून AI एजंट्सना सामग्री शोधून संवाद साधण्यास सक्षम करते.

## शिकण्याचे उद्दिष्ट

• AI एजंट्सच्या संदर्भात MCP, A2A, आणि NLWeb चा मुख्य उद्देश आणि फायदे ओळखणे.

• प्रत्येक प्रोटोकॉल कसा LLM, उपकरणे आणि इतर एजंट्स दरम्यान संवाद आणि परस्परसंवाद सुलभ करतो हे स्पष्ट करणे.

• जटिल एजेंटिक सिस्टम बांधण्यात प्रत्येक प्रोटोकॉलची वेगळी भूमिका ओळखणे.

## मॉडेल कॉन्टेक्स्ट प्रोटोकॉल

**मोॅडेल कॉन्टेक्स्ट प्रोटोकॉल (MCP)** हा एक खुला मानक आहे जो अनुप्रयोगांना LLMs ला संदर्भ आणि उपकरणे standardized मार्गाने देतो. यामुळे AI एजंट्स एका सुसंगत पद्धतीने वेगवेगळ्या डेटा स्रोतांशी आणि उपकरणांशी कनेक्ट होण्यासाठी "सार्वत्रिक अ‍ॅडॉप्टर" सारखा कार्य करू शकतो.

चला पाहूया MCP चे घटक, थेट API वापराच्या तुलनेत फायदे आणि AI एजंट्स कशी MCP सर्व्हर वापरू शकतात याचे उदाहरण.

### MCP चे मूळ घटक

MCP **क्लायंट-सर्व्हर आर्किटेक्चर** वर चालते आणि त्याचे मुख्य घटक आहेत:

• **होस्ट्स** म्हणजे LLM अनुप्रयोग (उदा. VSCode सारखा कोड संपादक) जे MCP सर्व्हरशी कनेक्शन सुरू करतात.

• **क्लायंट्स** म्हणजे होस्ट अनुप्रयोगातील घटक जे सर्व्हरशी एक-ते-एक कनेक्शन राखतात.

• **सर्व्हर्स** म्हणजे हलक्या वजनाचे कार्यक्रम जे विशिष्ट क्षमता उघड करतात.

प्रोटोकॉलमध्ये तीन मुख्य मूळ घटक आहेत जे MCP सर्व्हरच्या क्षमता आहेत:

• **टूल्स**: AI एजंट वापरू शकणारी वेगळ्या क्रियाकलापांची किंवा फंक्शन्सची यादी. उदा., हवामान सेवा "get weather" टूल देऊ शकते, किंवा e-commerce सर्व्हर "purchase product" टूल देऊ शकतो. MCP सर्व्हर प्रत्येक टूलचे नाव, वर्णन आणि इनपुट/आउटपुट यादी प्रदान करतो.

• **रिसोर्सेस**: हे केवळ वाचता येणारे डेटा आयटम्स किंवा दस्तऐवज आहे जे MCP सर्व्हर पुरवू शकतो, आणि क्लायंट्स त्यांना मागणीवर मिळवू शकतात. उदा., फाइल सामग्री, डेटाबेस नोंदी, किंवा लॉग फाइल्स. रिसोर्सेस टेक्स्ट (कोड किंवा JSON) किंवा बायनरी (प्रतिमा किंवा PDF) स्वरूपात असू शकतात.

• **प्रॉम्प्ट्स**: पूर्वनिर्धारित साच्यांचा संच आहे जे सुचवलेल्या प्रॉम्प्टसह अधिक जटिल वर्कफ्लोज सुलभ करतात.

### MCP चे फायदे

MCP AI एजंट्ससाठी महत्त्वाचे फायदे देतो:

• **डायनॅमिक टूल शोध**: एजंट्स एका सर्व्हरकडून उपलब्ध टूल्सची यादी आणि त्यांचे वर्णन डायनॅमिकली मिळवू शकतात. पारंपरिक API प्रणालीमध्ये एकदा कोडिंग करावी लागते जी API बदलल्यास पुनरावृत्ती करावी लागते. MCP मध्ये "एकदाच इंटिग्रेट करा" ही पद्धत वापरायची आहे, ज्यामुळे जास्त अनुकूलता मिळते.

• **LLMs मधील इंटरऑपरेबिलिटी**: MCP विविध LLMs सोबत कार्य करू शकतो, जेणेकरून मुख्य मॉडेल बदलून अधिक चांगले प्रदर्शन तपासता येते.

• **मानकीकृत सुरक्षा**: MCP मध्ये एक मानकीकृत प्रमाणीकरण पद्धत आहे, ज्यामुळे MCP सर्व्हर्सना ऍक्सेस वाढवण्याचे स्केलेबिलिटी सुधारते. हे पारंपरिक API च्या तुलना सोपे आहे.

### MCP उदाहरण

![MCP Diagram](../../../translated_images/mr/mcp-diagram.e4ca1cbd551444a1.webp)

विचारा की वापरकर्ता AI सहाय्यक वापरून फ्लाइट बुक करू इच्छितो, ज्यामध्ये MCP वापरलेले आहे.

1. **कनेक्शन**: AI सहाय्यक (MCP क्लायंट) एखाद्या विमानतळाच्या MCP सर्व्हरशी कनेक्ट करतो.

2. **टूल शोध**: क्लायंट विमानतळाच्या MCP सर्व्हरला विचारतो, "तुमच्याकडे कोणती टूल्स आहेत?" सर्व्हर "search flights" आणि "book flights" सारखे टूल्स परत पाठवतो.

3. **टूल आमंत्रण**: वापरकर्ता AI सहाय्यकाला म्हणतो, "पोर्टलँड ते होनोलुलूची फ्लाइट शोधा." AI सहाय्यक त्यासाठी "search flights" टूल कॉल करतो आणि संबंधित पॅरामीटर्स MCP सर्व्हरला देतो.

4. **अंमलबजावणी आणि प्रतिसाद**: MCP सर्व्हर विमानतळाच्या अंतर्गत बुकिंग API ला कॉल करतो, नंतर फ्लाइट माहिती (उदा., JSON डेटा) AI सहाय्यकाला पाठवतो.

5. **अधिक संवाद**: AI सहाय्यक फ्लाइट पर्याय सादर करतो. वापरकर्ता फ्लाइट निवडल्यानंतर, सहाय्यक "book flight" टूल वापरून बुकिंग पूर्ण करतो.

## एजंट-टू-एजंट प्रोटोकॉल (A2A)

MCP LLMs ना टूल्सशी जोडण्यावर लक्ष केंद्रित करतो, तर **एजंट-टू-एजंट (A2A) प्रोटोकॉल** AI एजंट्समध्ये संवाद आणि सहकार्य वाढवतो. A2A वेगवेगळ्या संस्थांतील, पर्यावरणातील आणि तंत्रज्ञानाचे वेगवेगळे स्टॅक्स असलेल्या AI एजंट्सना एकत्र आणून सामायिक कार्य पूर्ण करू शकतो.

चला A2A चे घटक, फायदे आणि आपल्या प्रवास अनुप्रयोगात कसा वापरला जाऊ शकतो हे पाहूया.

### A2A चे मूळ घटक

A2A एजंट्समध्ये संवाद सक्षम करतो आणि ते वापरकर्त्याच्या उपकार्यांवर एकत्र काम करतात. प्रोटोकॉलचे घटक:

#### एजंट कार्ड

MCP सर्व्हरप्रमाणेच, एजंट कार्डमध्ये:

- एजंटचे नाव.

- **सामान्य कार्यांचे वर्णन**.

- **विशिष्ट कौशल्यांची यादी** जे इतर एजंट्सना (किंवा मानवी वापरकर्त्यांना) समजावून सांगतात की कधी आणि का त्या एजंटकडे कॉल करावा.

- एजंटचा **सध्याचा एंडपॉइंट URL**.

- एजंटची **आवृत्ती** आणि **क्षमता** जसे की स्ट्रीमिंग प्रतिसाद आणि पुश नोटिफिकेशन्स.

#### एजंट एक्झिक्युटर

एजंट एक्झिक्युटर वापरकर्त्याच्या चॅटची संदर्भ माहिती दूरच्या एजंटला पाठवण्याची जबाबदारी पार पाडतो. दूरचा एजंट त्याच्या LLM चा वापर करून येणाऱ्या विनंत्यांचे विश्लेषण करतो आणित्याच्या अंतर्गत टूल्स वापरून कार्य पार पाडतो.

#### आर्टिफॅक्ट

दूरचा एजंट कार्य पूर्ण केल्यावर त्याचे उत्पादन आर्टिफॅक्ट म्हणून तयार होते. आर्टिफॅक्टमध्ये एजंटच्या कार्याचा परिणाम, पूर्ण झालेल्या कामाचे वर्णन आणि प्रोटोकॉलमधील पाठवलेला टेक्स्ट संदर्भ असतो. आर्टिफॅक्ट पाठवल्यानंतर दूरचा एजंटशी कनेक्शन बंद होते जोपर्यंत पुन्हा गरज भासत नाही.

#### इव्हेंट क्यू

ही गोष्ट अपडेट्स हाताळण्यासाठी आणि संदेश पाठवण्यासाठी वापरली जाते. एजंट-टू-एजंट सिस्टम्समध्ये ही उत्पादनासाठी विशेषतः महत्त्वाची आहे, कारण काम पूर्ण होईपर्यंत एजंट्समधील कनेक्शन बंद होऊ नये, विशेषत: जेव्हा काम पूर्ण होण्यास वेळ लागतो.

### A2A चे फायदे

• **वाढलेले सहकार्य**: वेगवेगळ्या विक्रेत्यांकडील आणि प्लॅटफॉर्मवरून एजंट्स परस्पर संवाद साधून एकत्र काम करू शकतात, जे पारंपरिक वेगळे असलेले सिस्टम्स मध्ये अखंड ऑटोमेशन सुकर करते.

• **मॉडेल निवडण्याची स्वातंत्र्य**: प्रत्येक A2A एजंट स्वतःच्या विनंत्यांसाठी कोणता LLM वापरायचा हे ठरवू शकतो, ज्यामुळे प्रत्येक एजंटसाठी ऑप्टिमाइझ किंवा फाइन-ट्यून मॉडेल वापरता येते, जे MCP च्या एका LLM कनेक्शनच्या तुलनेत अधिक लवचीक आहे.

• **निर्मित प्रमाणीकरण**: प्रमाणीकरण A2A प्रोटोकॉलमध्ये थेट समाविष्ट आहे, जे एजंट संवादासाठी सक्षम सुरक्षा फ्रेमवर्क प्रदान करतो.

### A2A उदाहरण

![A2A Diagram](../../../translated_images/mr/A2A-Diagram.8666928d648acc26.webp)

आमच्या प्रवासाच्या बुकिंग संकल्पनेला पुढे घेऊया, पण यावेळी A2A वापरून.

1. **वापरकर्त्याचा मल्टी-एजंटकडे विनंती**: वापरकर्ता "Travel Agent" A2A क्लायंट/एजंटशी संवाद साधतो, जसं, "कृपया पुढच्या आठवड्यासाठी होनोलुलूला संपूर्ण ट्रिप बुक करा - फ्लाइट्स, हॉटेल आणि रेंटल कार सहित".

2. **ट्रॅव्हल एजंटने नियोजन केले**: ट्रॅव्हल एजंट ही गुंतागुंतीची विनंती प्राप्त करतो. त्याचा LLM विचार करून तो इतर खास एजंट्सशी संवाद साधण्याची गरज ओळखतो.

3. **एजंट-एजंट संवाद**: ट्रॅव्हल एजंट A2A प्रोटोकॉलचा वापर करुन downstream एजंट्सशी (जसे की "Airline Agent," "Hotel Agent," "Car Rental Agent") कनेक्ट होतो, जे वेगवेगळ्या कंपन्यांनी तयार केलेले असू शकतात.

4. **कार्यांचे विनियोजन**: ट्रॅव्हल एजंट हे प्रत्येक विशेष एजंटला खास काम देतो (उदा., "होनोलुलूला फ्लाइट शोधा," "हॉटेल बुक करा," "कार भाड्याने घ्या"). हे एजंट्स स्वतःचे LLM आणि त्यांचे टूल्स वापरून आपापले काम पूर्ण करतात.

5. **एकत्रिकृत प्रतिसाद**: सर्व downstream एजंट्स काम पूर्ण केल्यानंतर, ट्रॅव्हल एजंट परिणाम (फ्लाइट तपशील, हॉटेल पुष्टी, कार बुकिंग) संकलित करुन वापरकर्त्यास एक आकर्षक चॅट स्वरुपात प्रतिसाद पाठवतो.

## नॅचरल लँग्वेज वेब (NLWeb)

संकेतस्थळे इंटरनेटवरील माहिती आणि डेटा वापरण्याचा प्रमुख मार्ग राहिले आहेत.

चला पाहूया NLWeb चे घटक, त्याचे फायदे आणि आमच्या प्रवास अनुप्रयोगाच्या उदाहरणातून NLWeb कसा कार्य करतो.

### NLWeb चे घटक

- **NLWeb अनुप्रयोग (कोर सेवा कोड)**: नैसर्गिक भाषा प्रश्न प्रक्रिया करणारी प्रणाली. हे प्लॅटफॉर्मच्या वेगवेगळ्या भागांना जोडून प्रतिसाद तयार करते. याला आपण संकेतस्थळाच्या नैसर्गिक भाषा वैशिष्ट्यांना चालना देणारे **इंजिन** मानू शकतो.

- **NLWeb प्रोटोकॉल**: संकेतस्थळासोबत नैसर्गिक भाषेचा संवाद साधण्यासाठी **मूलभूत नियमावली**. JSON स्वरूपात (बर्‍याचदा Schema.org वापरून) प्रतिसाद पाठवतो. याचा उद्देश "AI वेब" साठी सोप्या पायाभूत सुविधा तयार करणे आहे, जसे की HTML ने ऑनलाइन दस्तऐवज शेअर करण्यास सुलभ केले.

- **MCP सर्व्हर (मॉडेल कॉन्टेक्स्ट प्रोटोकॉल एंडपॉइंट)**: प्रत्येक NLWeb सेटअप ही MCP सर्व्हर म्हणूनही कार्य करतो. म्हणजे AI प्रणालींसोबत **टूल्स (जसे "ask" पद्धत) आणि डेटा** शेअर करू शकतो. यामुळे संकेतस्थळाची सामग्री AI एजंट्स वापरू शकतात आणि संकेतस्थळ मोठ्या "एजंट पर्यावरणाचा" भाग बनत आहे.

- **एम्बेडिंग मॉडेल्स**: हे मॉडेल संकेतस्थळाच्या सामग्रीला संख्यात्मक प्रतिनिधित्वांमध्ये (व्हेक्टर किंवा एम्बेडिंग्स मध्ये) रूपांतरित करतात. हे व्हेक्टर अर्थपूर्ण असतात जे संगणक तुलनेने आणि शोधण्यासाठी वापरतात. ते खास डेटाबेसमध्ये संग्रहित होतात. वापरकर्त्यांना कोणते एम्बेडिंग मॉडेल वापरायचे ते निवडता येते.

- **व्हेक्टर डेटाबेस (रिट्रीवल मेकॅनिझम)**: संकेतस्थळाच्या सामग्रीच्या एम्बेडिंग्ससाठी संग्रहस्थान. जेव्हा कोणीतरी प्रश्न विचारतो, तेव्हा NLWeb या डेटाबेसमध्ये सर्वात संबंधित माहिती जलद शोधतो. संभाव्य उत्तरांची यादी similarity नुसार क्रमवार देते. NLWeb Qdrant, Snowflake, Milvus, Azure AI Search, Elasticsearch सारख्या वेगवेगळ्या व्हेक्टर साठवणूक प्रणालींसोबत काम करतो.

### NLWeb उदाहरणाने

![NLWeb](../../../translated_images/mr/nlweb-diagram.c1e2390b310e5fe4.webp)

पुन्हा आमच्या प्रवास बुकिंग संकेतस्थळाकडे वळूया, पण यावेळी ते NLWeb द्वारे चालते.

1. **डेटा इनगेस्टशन**: प्रवास संकेतस्थळाचे विद्यमान उत्पादन सूची (उदा. फ्लाइट लिस्टिंग, हॉटेल वर्णन, टूर पॅकेजेस) Schema.org ने फॉरमॅट केले जातात किंवा RSS फीडद्वारे लोड केले जातात. NLWeb टूल्स या संरचित डेटाचे एम्बेडिंग तयार करतात आणि स्थानिक किंवा रिमोट व्हेक्टर डेटाबेसमध्ये साठवतात.

2. **नैसर्गिक भाषा प्रश्न (मानवी)**: वापरकर्ता संकेतस्थळाला भेट देतो आणि मेनू शोधण्याऐवजी चॅट इंटरफेस मध्ये टाईप करतो: "मला होनोलुलूमध्ये कुटुंबासाठी योग्य अशा, पुढच्या आठवड्याला स्विमिंग पूल असलेले हॉटेल शोधा".

3. **NLWeb प्रक्रिया**: NLWeb अनुप्रयोग हा प्रश्न प्राप्त करतो. तो प्रश्न समजून घेण्यासाठी LLM कडे पाठवतो आणि त्याच वेळी त्याच्या व्हेक्टर डेटाबेसमध्ये संबंधित हॉटेल लिस्टिंग शोधतो.

4. **अचूक परिणाम**: LLM डेटाबेस मधून शोधलेले निकाल समजावून सांगतो, "कुटुंबासाठी योग्य," "स्विमिंग पूल," आणि "होनोलुलू" या निकषांनुसार सर्वोत्तम जुळणी ओळखतो, नंतर नैसर्गिक भाषा प्रतिसाद तयार करतो. महत्त्वाची गोष्ट म्हणजे प्रतिसाद खऱ्या हॉटेल्सना संदर्भित करतो, बनावट माहिती नाही.

5. **AI एजंट परस्परसंवाद**: NLWeb MCP सर्व्हर म्हणून कार्य करतो, त्यामुळे बाह्य AI प्रवास एजंट देखील या संकेतस्थळाच्या NLWeb इंस्टन्सशी कनेक्ट होऊ शकतो. AI एजंट `ask` MCP पद्धती वापरुन वेबसाइटला थेट प्रश्न विचारू शकतो: `ask("Are there any vegan-friendly restaurants in the Honolulu area recommended by the hotel?")`. NLWeb हा प्रश्न प्रक्रिया करतो, त्याच्या रेस्टॉरंट डेटाबेस (जर लोड केलेला असेल) वापरतो आणि संरचित JSON प्रतिसाद देतो.

### MCP/A2A/NLWeb विषयी अधिक प्रश्न आहेत?

इतर शिकणाऱ्यांशी भेटण्यासाठी, कार्यालयीन वेळेत सहभागी होण्यासाठी आणि तुमचे AI एजंट्सचे प्रश्न विचारण्यासाठी [Microsoft Foundry Discord](https://aka.ms/ai-agents/discord) ला सामील व्हा.
