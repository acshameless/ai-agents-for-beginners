# Agentic Protocols ကို အသုံးပြုခြင်း (MCP, A2A and NLWeb)

[![Agentic Protocols](../../../translated_images/my/lesson-11-thumbnail.b6c742949cf1ce2a.webp)](https://youtu.be/X-Dh9R3Opn8)

> _(ဤသင်ခန်းစာ၏ ဗီဒီယိုကို ကြည့်ဖို့ အထက်ပါ ဓာတ်ပုံကို နှိပ်ပါ)_

AI agent များ၏ အသုံးပြုမှု တိုးလာ伴်အရ စံသတ်မှတ်ချက်၊ လုံခြုံရေးနှင့် ဖွင့်လှစ်ထားသော ဆန်းသစ်တီထွင်မှုကို ထောက်ပံ့ပေးနိုင်သော ပရိုတိုကောများလိုအပ်လာသည်။ ဒီသင်ခန်းစာတွင် အဆိုပါလိုအပ်ချက်ကို ဖြည့်ဆည်းရန် ရည်ရွယ်ထားသော ပရိုတိုကော ၃ မျိုး — Model Context Protocol (MCP), Agent to Agent (A2A) နှင့် Natural Language Web (NLWeb) — အကြောင်းကို လေ့လာပါမည်။

## နိဒါန်း

ဤသင်ခန်းစာတွင် ကျွန်ုပ်တို့က ဖော်ပြပါများကို လေ့လာပါမည်။

• **MCP** သည် AI Agent များအား အသုံးပြုသူ အလုပ်များကို ပြီးမြောက်စေရန် ပြင်ပကိရိယာများနှင့် ဒေတာများကို ထိမိရရန် ဘယ်လိုခွင့်ပြုသည်ကို။

• **A2A** သည် ကွဲပြားသော AI agent များအကြား ဆက်သွယ်ဆောင်ရွက်မှုနှင့် ပူးပေါင်းဆောင်ရွက်မှုကို မည်သို့ အထောက်အကူပြုသည်ကို။

• **NLWeb** သည် မည်သည့် website မဆို သဘာဝဘာသာ စကား အင်တာဖေ့စ်များကို ပေးနိုင်စေခြင်းအားဖြင့် AI agent များအနေဖြင့် ဝက်ဘ်ဆိုင် content ကို ရှာဖွေ ထိတွေ့ ဆက်ဆံနိုင်ရန် မည်သို့ အကူဖြစ်စေသည်ကို။

## သင်ယူရမည့်ရည်ရွယ်ချက်များ

• **သိရှိရန်** MCP, A2A, နှင့် NLWeb တို့၏ အဓိက ရည်ရွယ်ချက်များနှင့် အကျိုးကျေးဇူးများကို AI agent များ၏ အ contexts ထဲတွင် ဖော်ထုတ်နိုင်ရန်။

• **ရှင်းပြရန်** တစ်ခုချင်းစီသော ပရိုတိုကောက LLM များ၊ ကိရိယာများနှင့် အခြား agent များအကြား ဆက်သွယ်မှုနှင့် အပြန်အလှန် ဆက်ဆံမှုကို မည်သို့ လွှမ်းမိုးပေးသည်ကို။

• **သိမြင်နိုင်ရန်** ခက်ခဲသော agentic စနစ်များ တည်ဆောက်ရာတွင် တစ်ခုချင်းစီ၏ သီးခြားတာဝန်များကို မှတ်ယူနိုင်ရန်။

## Model Context Protocol

Model Context Protocol (MCP) သည် လျှပ်တပြတ်မဖြစ်ဘဲ application များမှ LLM များထံ context နှင့် ကိရိယာများကို စံညွှန်းထားသော နည်းလမ်းဖြင့် ပေးပို့နိုင်စေသော ဖွင့်လှစ်စံချိန်တစ်ခုဖြစ်သည်။ ၎င်းသည် AI Agent များအနေဖြင့် မတူညီသော ဒေတာအရင်းမြစ်များနှင့် ကိရိယာများကို တစ်ဘက်တည်းက ဆက်သွယ်နိုင်ရန် "ယေဘုယျ အပြောင်းလဲရလွယ် adaptor" တစ်ခုလို ပြုလုပ်ပေးသည်။

MCP ၏ အစိတ်အပိုင်းများ၊ တိုက်ရိုက် API အသုံးပြုမှုနှင့်နှိုင်းယှဉ်၍ အကျိုးကျေးဇူးများ နှင့် AI agent များ MCP server ကို မည်သို့ အသုံးပြုနိုင်သည်ကို ဥပမာတစ်ခုဖြင့် ကြည့်ကြပါစို့။

### MCP ၏ အဓိက အစိတ်အပိုင်းများ

MCP သည် **client-server architecture** အပေါ် မူတည်၍ အဓိက အစိတ်အပိုင်းများမှာ -

• **Hosts** သည် MCP Server ဆက်သွယ်မှုများကို စတင်ဖြစ်စေသည့် LLM application များ (ဥပမာ VSCode စသည်) ဖြစ်သည်။

• **Clients** သည် host application အတွင်းရှိ component များဖြစ်ပြီး server များနှင့် တစ်ဦးချင်း ဆက်သွယ်မှုကို ထိန်းသိမ်းထားသည့် အစိတ်အပိုင်းများဖြစ်သည်။

• **Servers** သည် အထူးစွမ်းဆောင်ရည်များကို ထုတ်ဖော်ပြသသည့် အလင်းမြန်သော program များဖြစ်သည်။

ပရိုတိုကောတွင် MCP Server ၏ စွမ်းဆောင်ရည်များဖြစ်သော core primitives သုံးခု ပါဝင်သည်။

• **Tools**: ၎င်းတို့သည် AI agent သည် တစ်ခုခု လုပ်ဆောင်ရန်ခေါ်ယူနိုင်သည့် ဖြတ်သန်းနိုင်သည့် အရေးယူချက်များ သို့မဟုတ် function များဖြစ်သည်။ ဥပမာ၊ ရာသီဥတု ဝန်ဆောင်မှုတစ်ခုသည် "get weather" tool ကို ထုတ်ဖော်နိုင်ပြီး e-commerce server တစ်ခုသည် "purchase product" tool ကို ထုတ်ဖော်နိုင်သည်။ MCP server များသည် တစ်ခုချင်း tool ၏ name, description နှင့် input/output schema များကို ၎င်းတို့၏ capabilities listing တွင် ကြော်ငြာပြသသည်။

• **Resources**: ၎င်းတို့သည် MCP server တစ်ခုမှ ပေးနိုင်သည့် ဖတ်-only ဒေတာအချက်အလက်များ သို့မဟုတ် စာရွက်စာတမ်းများဖြစ်ပြီး client များ ကာလတိုင်း လိုအပ်သလို ရယူနိုင်သည်။ ဥပမာများတွင် ဖိုင်အကြောင်းအရင်းအချက်အလက်များ၊ ဒေတာဘေ့စ် မှတ်တမ်းများ သို့မဟုတ် log ဖိုင်များ ပါဝင်သည်။ Resources များသည် စာသား (code သို့မဟုတ် JSON ကဲ့သို့) သို့မဟုတ် binary (ပုံများ သို့မဟုတ် PDF များ) ဖြစ်နိုင်သည်။

• **Prompts**: ၎င်းတို့သည် ကြိုတင်သတ်မှတ်ထားသော template များဖြစ်ပြီး ပိုမို ပြည့်စုံသော workflow များအတွက် အကြံပြု prompt များကို ပေးသည်။

### MCP ၏ အကျိုးကျေးဇူးများ

MCP သည် AI Agent များအတွက် အရေးကြီးသော အားသာချက်များကို ပေးသည် -

• **Dynamic Tool Discovery**: Agent များသည် server မှ လက်ရှိ အသုံးပြုနိုင်သော tool များ၏ စာရင်းကို အကြောင်းအရာပါ ဖော်ပြချက်များနှင့်အတူ dynamic ဖြစ်စွာ လက်ခံရရှိနိုင်သည်။ ၎င်းမှာ ပုံမှန် API များကဲ့သို့ integration များအတွက် static coding လိုအပ်သည်၊ API ပြင်ဆင်မှုတစ်ခုချက်တိုင်း code ပြင်ဆင်ရမည့် အခြေအနေကို မလိုအပ်ဘဲ "တစ်ကြိမ် အတန်းချုပ်ချင်းပိုင်း" ဖော်မြူလာကို ပေးသည်။ MCP သည် ပိုမို သေချာသော အပြောင်းလဲမှုကို ချဉ်းကပ်စေသည်။

• **Interoperability Across LLMs**: MCP သည် မတူညီသော LLM များအတွင်း လည်ပတ်နိုင်ပြီး core model များကို အလွယ်တကူ ပြောင်းလဲ၍ ပိုမိုကောင်းမွန်သော စွမ်းဆောင်ရည်များကို စစ်ဆေးနိုင်သည်။

• **Standardized Security**: MCP တွင် စံသတ်မှတ် authentication နည်းလမ်းတစ်ခု ပါဝင်ပြီး MCP server အသစ်များ ထပ်မံ ထည့်သွင်းသောအခါ scalability ကို တိုးမြှင့်ပေးသည်။ ၎င်းသည် ပုံမှန် API များအတွက် အမျိုးမျိုးသော key များနှင့် authentication အမျိုးအစားများကို စီမံရသော ပိုမိုရှုပ်ထွေးမှုကို လျော့ချပေးသည်။

### MCP ဥပမာ

![MCP Diagram](../../../translated_images/my/mcp-diagram.e4ca1cbd551444a1.webp)

တစ်ယောက်အသုံးပြုသူသည် MCP ဖြင့် စွမ်းအားပေးထားသော AI အကူအညီတစ်ခုကို အသုံးပြု၍ လေယာဉ်တန်းခရီးဘွတ်ကင်ချဖို့ လိုချင်ကြောင်း စဉ်းစားကြည့်ပါ။

1. **ဆက်သွယ်မှု**: AI အကူအညီ (MCP client) သည် လေကြောင်းလိုင်းမှ ပေးထားသည့် MCP server သို့ ဆက်သွယ်သည်။

2. **Tool Discovery**: client သည် လေကြောင်းလိုင်း၏ MCP server ကို "ဘယ် tool များ ရှိသနည်း?" ဟု မေးသည်။ server သည် "search flights" နှင့် "book flights" ကဲ့သို့သော tool များကို ပြန်စာပေးမည်။

3. **Tool Invocation**: သင်သည် AI အကူအညီထံ "Portland မှ Honolulu သို့ ခရီးစဉ်ကို ရှာပေးပါ" ဟု တောင်းဆိုသည်။ AI အကူအညီသည် ၎င်း၏ LLM ကို အသုံးပြု၍ "search flights" tool ကို ခေါ်ရန် လိုအပ်သည်ကို သတ်မှတ်ကာ အဆိုပါ parameter များ (origin, destination) ကို MCP server သို့ ပို့သည်။

4. **အကောင်အထည်ဖော်ခြင်းနှင့် တုံ့ပြန်ချက်**: MCP server သည် wrapper အဖြစ် လုပ်ဆောင်ကာ လေကြောင်းလိုင်း၏ internal booking API ကို အမှန်တကယ် ခေါ်ဆောင်သွားပြီး လေယာဉ်သတင်းအချက်အလက်များ (ဥပမာ JSON data) ကို လက်ခံရရှိကာ AI အကူအညီထံ ပေးပို့သည်။

5. **နောက်ထပ်ဆက်သွယ်မှု**: AI အကူအညီသည် လေယာဉ်ရွေးချယ်စရာများကို တင်ပြသည်။ သင်သည် လေယာဉ်တစ်စီးကို ရွေးချယ်လျှင် အကူအညီသည် ထပ်မံ "book flight" tool ကို အဆိုပါ MCP server ပေါ်တွင် ခေါ်၍ booking ကို ပြီးမြောက်စေနိုင်သည်။

## Agent-to-Agent Protocol (A2A)

MCP သည် LLM များနှင့် ကိရိယာများကို ဆက်သွယ်ပေးရန် အာရုံစိုက်သော်လည်း၊ Agent-to-Agent (A2A) protocol သည် ကွဲပြားသော AI agent များအကြား ဆက်သွယ်ဆောင်ရွက်မှုနှင့် ပူးပေါင်းဆောင်ရွက်မှုကို ဖော်ဆောင်ပေးသည်။ A2A သည် အဖွဲ့အစည်းများ၊ ပတ်ဝန်းကျင်များနှင့် နည်းပညာ စတက်များ များစွာအတွက် agent များကို ဆက်သွယ်ပေးကာ ပူးပေါင်းအလုပ်လုပ်ခြင်းဖြင့် ပြည့်စုံသော တာဝန်တစ်ခုကို ပြီးမြောက်စေသည်။

ခရီးသွားလျှောက် application ကို ဥပမာယူကာ A2A ၏ အစိတ်အပိုင်းများနှင့် အကျိုးကျေးဇူးများကို ကြည့်ကြရအောင်။

### A2A ၏ အဓိက အစိတ်အပိုင်းများ

A2A သည် agent များအကြား ဆက်သွယ်မှုနှင့် သေချာစွာ subtask များကို ပေးပို့ကာ အတူတကွ အလုပ်လုပ်စေခြင်းကို အဓိကထားသည်။ ပရိုတိုကော၏ တစ်ခုချင်းစီသော component များသည် ၎င်းကို ပံ့ပိုးပေးသည်။

#### Agent Card

MCP server သည် tool စာရင်းကို မျှဝေသည့်နည်းနည်းနှင့် ဆင်တူစွာ Agent Card သည် -
- Agent ၏ Name။
- ၎င်း ပြီးမြောက်ပေးနိုင်သည့် အထွေထွေ အလုပ်များ၏ **ဖော်ပြချက်**။
- အခြား agent များ (သို့မဟုတ် လူအသုံးပြုသူများ) ၎င်း agent ကို ဘယ်အချိန်နှင့် အဘယ်ကြောင့် ခေါ်သင့်သည်ကို နားလည်နိုင်စေရန် **သတ်မှတ်ထားသော ကျွမ်းကျင်မှု ရှိ/မရှိ စာရင်း** နှင့် ဖော်ပြချက်များ။
- Agent ၏ **လက်ရှိ Endpoint URL**။
- Agent ၏ **version** နှင့် **capabilities** (ဥပမာ streaming responses, push notifications) များ။

#### Agent Executor

Agent Executor ၏တာဝန်မှာ **အသုံးပြုသူ chat ၏ context ကို remote agent ထံ ပို့ဆောင်ပေးခြင်း** ဖြစ်သည်။ remote agent သည် ပြီးမြောက်ရမည့် တာဝန်ကို နားလည်ရန် ၎င်းကို လိုအပ်သည်။ A2A server တစ်ခုတွင် agent တစ်ခုသည် ၎င်း၏ ကိုယ်ပိုင် LLM ကို အသုံးပြုကာ ဝင်လာသော တောင်းဆိုမှုများကို parse ပြီး ကိုယ်ပိုင် internal tools များဖြင့် တာဝန်များကို အကောင်အထည်ဖော်သည်။

#### Artifact

remote agent မှ တောင်းဆိုထားသည့် တာဝန်ကို ပြီးမြောက်သွားလျှင် ၎င်း၏ အလုပ်ထုတ်ကုန်ကို artifact အဖြစ် ဖန်တီးသည်။ artifact သည် **agent ၏ အလုပ်ရလဒ်**, **ဘာကို ပြီးမြောက်ခဲ့သည်ဆိုသည်၏ ဖော်ပြချက်**, နှင့် ပရိုတိုကောမှတဆင့် ပို့ပေးသော **စာသား context** ကို ထည့်သွင်းထားသည်။ artifact ကို ပို့ပြီးနောက် remote agent သို့ ဆက်သွယ်မှုကို ပြန်ပိတ်လိုက်ပြီး ထပ်မံ လိုအပ်သည့်အခါ သာ ဖွင့်မည်။

#### Event Queue

ဤ component သည် **အပ်ဒိတ်များကို ကိုင်တွယ်ခြင်းနှင့် ขောမက်များ ပို့ဆောင်ခြင်း** အတွက် အသုံးပြုသည်။ production ခေတ်တွင် agentic system များအတွက် အထူးစိတ်တိုင်းရင်းဆုံးဖြစ်သည် — တာဝန်များပြီးမြောက်ရန် ကြာမြင့်နိုင်သည့်အခါ agent များအကြား ဆက်သွယ်မှုကို ပြတ်တောက်ခြင်းမှ ကာကွယ်ရန် အရေးကြီးသည်။

### A2A ၏ အကျိုးကျေးဇူးများ

• **ပူးပေါင်းဆောင်ရွက်မှု မြှင့်တင်ခြင်း**: ကွဲပြားသော vendor များနှင့် platform များမှ agent များကို အချင်းချင်း ဆက်သွယ်၊ context မျှဝေ၊ နှင့် ပူးပေါင်း အလုပ်လုပ်ပေးနိုင်စေသည်။ ယင်းသည် ရိုးရိုး ခွဲထားသော စနစ်များအတွင်း automation ကို ချိတ်ဆက်ပေးသည်။

• **Model ရွေးချယ်မှုဖြစ်နိုင်ခြင်း**: တစ်ခုချင်း A2A agent တစ်ခုချင်းစီသည် ၎င်း၏ request များကို စီမံရန် မည်သည့် LLM ကို အသုံးပြုမည့်အကြောင်း ကို ကိုယ်တိုင်ဆုံးဖြတ်နိုင်သည်၊ ထို့ကြောင့် agent တစ်ခုချင်းစီအတွက် အ tốiွန်ကျပ် သို့မဟုတ် fine-tuned model များကို သတ်မှတ်နိုင်သည်၊ MCP သာမန်အချို့တွင်ရှိသည့် တစ်ခုတည်းသော LLM ဆက်သွယ်မှုနှင့် မတူပါ။

• **Authentication built-in**: Authentication သည် A2A ပရိုတိုကောအတွင်း တိုက်ရိုက် တွဲဖက်ထားပြီး agent များအကြား အပြန်အလှန် ဆက်သွယ်မှုများအတွက် သက်တမ်းရှည် လုံခြုံမှု အခြေခံကို ပေးသည်။

### A2A ဥပမာ

![A2A Diagram](../../../translated_images/my/A2A-Diagram.8666928d648acc26.webp)

ကျွန်ုပ်တို့၏ ခရီးသွား ဘွတ်ကင် အခြေအနေကို A2A အသုံးပြု၍ တိုးမြှင့်ကြည့်ရအောင်။

1. **အသုံးပြုသူ တောင်းဆိုချက်မှ Multi-Agent သို့**: အသုံးပြုသူတစ်ဦးသည် "Travel Agent" A2A client/agent နှင့် ဆက်သွယ်ကာ, "လာမည့် အပတ်တွင် Honolulu သို့ စုစုပေါင်း ခရီးစဉ် တစ်ခုကို စီစဉ်ပေးပါ — လေယာဉ်၊ ဟိုတယ်၊ နှင့် ငှားကားလည်းပါပါစေ" ဟု တောင်းဆိုနိုင်သည်။

2. **Travel Agent မှ အစီအစဉ်ချခြင်း**: Travel Agent သည် ဤရှုပ်ထွေးသော တောင်းဆိုချက်ကို လက်ခံသည်။ ၎င်း၏ LLM ကို အသုံးပြုကာ တာဝန်ကို ယာယီခွဲခြမ်းစိတ်ဖြာသောအခါ ၎င်းသည် အထူးကျွမ်းကျင် agent များကို ဆက်သွယ်ရန်လိုအပ်ကြောင်း ဆုံးဖြတ်သည်။

3. **Agent များအကြား ဆက်သွယ်မှု**: Travel Agent သည် A2A ပရိုတိုကောကို အသုံးပြုကာ downstream agent များ (ဥပမာ "Airline Agent", "Hotel Agent", "Car Rental Agent") သို့ ချိတ်ဆက်သည်။ ၎င်းတို့ကို ကွဲပြားသည့် ကုမ္ပဏီများက ဖန်တီးထားနိုင်သည်။

4. **တာဝန်များ ကို ရက်ရွှေ့ပေးခြင်း**: Travel Agent သည် ဤ အထူးကျွမ်းကျင် agent များသို့ အထူးတာဝန်များကို ပို့သည် (ဥပမာ "Find flights to Honolulu", "Book a hotel", "Rent a car")။ ၎င်းတို့၏ ကိုယ်ပိုင် LLM များနှင့် ကိုယ်ပိုင် ကိရိယာများ (MCP server များ ဖြစ်နိုင်သည်) ကို အသုံးပြုကာ တစ်ဦးချင်းစီသည် ၎င်းတို့၏ တာဝန်ကို အကောင်အထည်ဖော်သည်။

5. **တစ်စုတစ်စည်းဖြင့် ပြန်လည်တင်ပြခြင်း**: downstream agent များအားလုံး လုပ်ငန်းများကို ပြီးမြောက်ထားလျှင် Travel Agent သည် ရလဒ်များ (လေယာဉ်အသေးစိတ်၊ ဟိုတယ် အတည်ပြုချက်၊ ငှားကား အကြောင်း) ကို စုပေါင်းပြီး အသုံးပြုသူထံ chat-style ဖြင့် ပြန်လည် တင်ပြပေးသည်။

## Natural Language Web (NLWeb)

ဝက်ဘ်ဆိုက်များသည် အင်တာနက်တစ်လျှောက် သတင်းအချက်အလက်နှင့် ဒေတာများကို အသုံးပြုသူများ အဓိက ဝင်ရောက် ရယူနိုင်သည့် နည်းလမ်း ဖြစ်ခဲ့သည်။

NLWeb ၏ အမျိုးမျိုးသော အစိတ်အပိုင်းများ၊ NLWeb ၏ အကျိုးကျေးဇူးများနှင့် ကျွန်ုပ်တို့၏ ခရီးသွားလျှောက် application တွင် NLWeb သည် မည်သို့ လုပ်ဆောင်သည်ကို ဥပမာဖြင့် ကြည့်ပါမည်။

### NLWeb ၏ အစိတ်အပိုင်းများ

- **NLWeb Application (Core Service Code)**: သဘာဝဘာသာမေးခွန်းများကို ကိုင်တွယ်ထိန်းချုပ်သည့် စနစ်။ ၎င်းသည် ပလက်ဖောင်း၏ အပိုင်းအစများကို ချိတ်ဆက်ကာ တုံ့ပြန်ချက်များ ဖန်တီးပေးသည်။ ၎င်းကို ဝက်ဘ်ဆိုက်၏ သဘာဝဘာသာ features များကို အားဖြည့်ပေးသည့် engine ဟု စဉ်းစားနိုင်သည်။

- **NLWeb Protocol**: ၎င်းသည် ဝက်ဘ်ဆိုက်နှင့် သဘာဝဘာသာ စကား အပြန်အလှန် ဆက်သွယ်ရန် မူလလမ်းညွှန် စည်းကမ်းများဖြစ်သည်။ ၎င်းသည် JSON ဖော်ပုံဖြင့် တုံ့ပြန်ချက်များ ပေးပို့သည် (အများအားဖြင့် Schema.org ကို အသုံးပြု၍)။ ၎င်း၏ ရည်ရွယ်ချက်မှာ HTML ကွန်ရက်ကဲ့သို့ “AI Web” အတွက် ရိုးရှင်းသော အခြေခံကို ဖန်တီးပေးရန် ဖြစ်သည်။

- **MCP Server (Model Context Protocol Endpoint)**: တစ်ခုချင်း NLWeb စနစ်သည် MCP server အဖြစ်လည်း လည်ပတ်သည်။ ၎င်းသည် အခြား AI စနစ်များနှင့် **tools (ဥပမာ “ask” method) နှင့် ဒေတာများကို မျှဝေနိုင်သည်**။ လက်တွေ့တွင် ၎င်းသည် ဝက်ဘ်ဆိုက်၏ အကြောင်းအရာနှင့် ကျွမ်းကျင်မှုများကို AI agent များအတွက် အသုံးပြုနိုင်စေသည်။ ထို့ကြောင့် ဝက်ဘ်ဆိုက်သည် "agent ecosystem" ကြီးထဲ တစ်စိတ်တစ်ပိုင်း ဖြစ်လာနိုင်သည်။

- **Embedding Models**: ၎င်းမျိုးက မော်ဒယ်များသည် ဝက်ဘ်ဆိုက် အကြောင်းအရာများကို ကွန်ပျူတာများ ရှာဖွေရေး နှင့် နှိုင်းယှဉ်နိုင်ရန် ရောင်ဖျို ဗက်တာ (vectors) များသို့ ပြောင်းလဲပေးသည်။ ဗက်တာများသည် အဓိပ္ပာယ်ကို ဖမ်းယူထားပြီး ရှာဖွေရန် အတွက် အသုံးပြုနိုင်သည်။ ၎င်းတို့ကို အထူး ဒေတာဘေ့စ်တစ်ခုတွင် သိမ်းဆည်းထားပြီး အသုံးပြုသူများသည် အသုံးပြုလိုသည့် embedding model ကို ရွေးချယ်နိုင်သည်။

- **Vector Database (Retrieval Mechanism)**: ဤဒေတာဘေ့စ်သည် **ဝက်ဘ်ဆိုက်အကြောင်းအရာ၏ embedding များကို စုဆောင်းထားသည်**။ မည်သူမဆို မေးခွန်းတစ်ခုတင်လာသည့်အခါ NLWeb သည် vector database ကို စိစစ်ကာ အရေးကြီးဆုံး သတင်းအချက်အလက်များကို လျင်မြန်စွာ ရှာဖွေသည်။ ၎င်းသည် ဆင်တူမှုအလိုက် အမြင့်ဆုံး အဖြေများကို အဆင့်သတ်မှတ်ပေးသည်။ NLWeb သည် Qdrant, Snowflake, Milvus, Azure AI Search, နှင့် Elasticsearch ကဲ့သို့သော vector storage systems များနှင့် လည်ပတ်နိုင်သည်။

### NLWeb ဥပမာ

![NLWeb](../../../translated_images/my/nlweb-diagram.c1e2390b310e5fe4.webp)

ကျွန်ုပ်တို့၏ ခရီးသွားဘွတ်ကင် website ကို ထပ်မံစဉ်းစားပါ၊ 이번에는 NLWeb ဖြင့် အားပေးထားသည်လို့ ယူဆပါ။

1. **ဒေတာ ထည့်သွင်းခြင်း**: ခရီးသွား ဝက်ဘ်ဆိုက်၏ ရှိပြီးသား product catalogs (ဥပမာ flight listings, hotel descriptions, tour packages) ကို Schema.org ဖြင့် ဖော်စပ်ထားသော ခံယူချက်များ သို့မဟုတ် RSS feed များမှတဆင့် load လုပ်သည်။ NLWeb ၏ ရည်မှန်းချက်က ဒေတာကို ingest လုပ်ကာ embedding များကို ဖန်တီး၍ ဒေသဆိုင်ရာ သို့မဟုတ် cloud vector database တစ်ခုတွင် သိမ်းဆည်းခြင်းဖြစ်သည်။

2. **သားနှင့်ဘာသာ ရှင်းလင်းမေးခွန်း (လူ)**: အသုံးပြုသူသည် ဝက်ဘ်ဆိုက်ကို ရောက်လာပြီး menu များကို ရှာဖွေနေရင်းမဟုတ်ဘဲ chat interface ထဲတွင် "လာမည့် အပတ်အတွက် မိသားစုအဆင်ပြေသည့် ဟိုတယ်တစ်ခုကို Honolulu တွင် ရှာပေးပါ — ရေကူးကန် ပါဝင်ရမည်" ဟု ရိုက်ထည့်နိုင်သည်။

3. **NLWeb ကိုင်တွယ်ခြင်း**: NLWeb application သည် ၎င်းမေးခွန်းကို လက်ခံသည်။ ၎င်းသည် မေးခွန်းကို ကို လေ့လာရန် LLM သို့ ပို့ပြီး တစ်ပြိုင်နက်တည်း vector database ကိုလည်း သက်ဆိုင်ရာ ဟိုတယ်စာရင်းများအတွက် ရှာဖွေမှုပြုလုပ်သည်။

4. **မှန်ကန်သော ရလဒ်များ**: LLM သည် database မှ ရှာဖွေမှုရလဒ်များကို အဓိပ္ပာယ်ဖော်ထုတ်ပေးကာ "မိသားစုအဆင်ပြေခြင်း", "ရေကူးကန်", "Honolulu" ကဲ့သို့သော မေးခွန်းများအပေါ် အကောင်းဆုံး ကိုက်ညီမှုများကို ระบุပြီး သဘာဝဘာသာ ဖြင့် တုံ့ပြန်ချက်ကို ဖော်ပြပေးသည်။ အရေးကြီးဆုံးမှာ ၎င်း၏ တုံ့ပြန်ချက်သည် ဝက်ဘ်ဆိုက်၏ ကက်တာလော့ဂ်မှ တကယ့် ဟိုတယ်များကို ကိုးကားပေးပြီး ဒေတာမရှိသော အချက်အလက်များ မပြောဖျက်ခြင်းဖြစ်စေသည်။

5. **AI Agent ၏ ဆက်သွယ်မှု**: NLWeb သည် MCP server အဖြစ်လည်း လည်ပတ်သောကြောင့် ပြင်ပ AI travel agent တစ်ခုလည်း ၎င်း၏ NLWeb instance သို့ ချိတ်ဆက်နိုင်သည်။ AI agent သည် `ask` MCP method ကို အသုံးပြုကာ တိုက်ရိုက် ဝက်ဘ်ဆိုက်ကို မေးမြန်းနိုင်သည် — ဥပမာ `ask("Are there any vegan-friendly restaurants in the Honolulu area recommended by the hotel?")`။ NLWeb instance သည် ၎င်းကို အလုပ်လုပ်ကာ restaurants အချက်အလက် database (load ထားပါက) ကို အသုံးပြုပြီး စနစ်တကျ JSON ဖောင်မာဖြင့် တုံ့ပြန်ချက် ပေးမည်။

### MCP/A2A/NLWeb အကြောင်း ထပ်မံ မေးခွန်းများ ရှိပါသလား။

အခြားသင်ယူလိုသူများနှင့် တွေ့ဆုံရန်၊ office hours များကို တက်ရောက်ရန်နှင့် သင်၏ AI Agents ဆိုင်ရာ မေးခွန်းများကို ဖြေရှင်းပေးစေရန် [Microsoft Foundry Discord](https://aka.ms/ai-agents/discord) ကို ဝင်ရောက်ပါ။
