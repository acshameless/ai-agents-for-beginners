# ਏਜੇਂਟਿਕ ਪ੍ਰੋਟੋਕੋਲਾਂ ਦੀ ਵਰਤੋਂ (MCP, A2A ਅਤੇ NLWeb)

[![ਏਜੇਂਟਿਕ ਪ੍ਰੋਟੋਕੋਲ](../../../translated_images/pa/lesson-11-thumbnail.b6c742949cf1ce2a.webp)](https://youtu.be/X-Dh9R3Opn8)

> _(ਉਪਰੋਕਤ ਚਿੱਤਰ 'ਤੇ ਕਲਿੱਕ ਕਰਕੇ ਇਸ ਪਾਠ ਦਾ ਵੀਡੀਓ ਵੇਖੋ)_

ਜਿਵੇਂ-जਿਵੇਂ AI ਏਜੰਟਾਂ ਦੀ ਵਰਤੋਂ ਵਧਦੀ ਜਾ ਰਹੀ ਹੈ, ਉਵੇਂ-ਉਵੇਂ ਐਸੇ ਪ੍ਰੋਟੋਕੋਲਾਂ ਦੀ ਲੋੜ ਵੱਧਦੀ ਹੈ ਜੋ ਮਿਆਰੀਕਰਨ, ਸੁਰੱਖਿਆ ਅਤੇ ਖੁੱਲ੍ਹੇ ਨਵੀਨਤਾ ਲਈ ਸਹਾਇਤਾ ਨੂੰ ਯਕੀਨੀ ਬਣਾਉਂਦੇ ਹਨ। ਇਸ ਪਾਠ ਵਿੱਚ, ਅਸੀਂ 3 ਪ੍ਰੋਟੋਕੋਲਾਂ ਨੂੰ ਕਵਰ ਕਰਾਂਗੇ ਜੋ ਇਸ ਲੋੜ ਨੂੰ ਪੂਰਾ ਕਰਨ ਦੀ ਕੋਸ਼ਿਸ਼ ਕਰ ਰਹੇ ਹਨ - Model Context Protocol (MCP), Agent to Agent (A2A) ਅਤੇ Natural Language Web (NLWeb)।

## ਪਰਿਚਯ

ਇਸ ਪਾਠ ਵਿੱਚ, ਅਸੀਂ ਇਹ ਸਮਗਰੀ ਕਵਰ ਕਰਾਂਗੇ:

• ਕਿਵੇਂ **MCP** AI ਏਜੰਟਾਂ ਨੂੰ ਯੂਜ਼ਰ ਟਾਸਕ ਪੂਰੇ ਕਰਨ ਲਈ ਬਾਹਰੀ ਟੂਲਾਂ ਅਤੇ ਡੇਟਾ ਤੱਕ ਪਹੁੰਚ ਦਿੰਦਾ ਹੈ।

• ਕਿਵੇਂ **A2A** ਵੱਖ-ਵੱਖ AI ਏਜੰਟਾਂ ਵਿਚਕਾਰ ਸੰਚਾਰ ਅਤੇ ਸਹਿਯੋਗ ਨੂੰ ਯੋਗ ਬਣਾਉਂਦਾ ਹੈ।

• ਕਿਵੇਂ **NLWeb** ਕਿਸੇ ਵੀ ਵੈੱਬਸਾਈਟ 'ਤੇ ਕੁਦਰਤੀ ਭਾਸ਼ਾ ਇੰਟਰਫੇਸ ਲਿਆਉਂਦਾ ਹੈ, ਜਿਸ ਨਾਲ AI ਏਜੰਟ ਸਮੱਗਰੀ ਨੂੰ ਖੋਜ ਅਤੇ ਇੰਟਰੈਕਟ ਕਰ ਸਕਦੇ ਹਨ।

## ਸਿੱਖਣ ਦੇ ਲਕਸ਼

• **ਪਛਾਣੋ** AI ਏਜੰਟਾਂ ਦੇ ਸੰਦਰਭ ਵਿੱਚ MCP, A2A, ਅਤੇ NLWeb ਦਾ ਮੁੱਖ ਉਦੇਸ਼ ਅਤੇ ਲਾਭ।

• **ਵਿਆਖਿਆ ਕਰੋ** ਕਿ ਹਰ ਪ੍ਰੋਟੋਕੋਲ ਕਿਵੇਂ LLMs, ਟੂਲਾਂ, ਅਤੇ ਹੋਰ ਏਜੰਟਾਂ ਵਿਚਕਾਰ ਸੰਚਾਰ ਅਤੇ ਇੰਟਰੈਕਸ਼ਨ ਨੂੰ ਸਹੂਲਤ ਦਿੰਦਾ ਹੈ।

• **ਪਛਾਣੋ** ਕਿ ਕੰਪਲੈਕਸ ਏਜੈਂਟਿਕ ਸਿਸਟਮ ਬਣਾਉਣ ਵਿੱਚ ਹਰੇਕ ਪ੍ਰੋਟੋਕੋਲ ਦੀ ਵੱਖਰੀ ਭੂਮਿਕਾ ਕੀ ਹੈ।

## ਮਾਡਲ ਕਾਂਟੈਕਸਟ ਪ੍ਰੋਟੋਕੋਲ

The **Model Context Protocol (MCP)** ਇੱਕ ਖੁੱਲ੍ਹਾ ਮਿਆਰੀਕਰਨ ਹੈ ਜੋ ਐਪਲੀਕੇਸ਼ਨਾਂ ਲਈ LLMs ਨੂੰ ਸੰਦਰਭ ਅਤੇ ਟੂਲ ਪ੍ਰਦਾਨ ਕਰਨ ਦਾ ਮਿਆਰੀ ਤਰੀਕਾ ਦਿੰਦਾ ਹੈ। ਇਹ ਇਕ "ਯੂਨੀਵਰਸਲ ਐਡੈਪਟਰ" ਦੀ ਤਰ੍ਹਾਂ ਕੰਮ ਕਰਨ ਦੇ ਯੋਗ ਬਣਾਉਂਦਾ ਹੈ ਜੋ ਵੱਖ-ਵੱਖ ਡੇਟਾ ਸ੍ਰੋਤਾਂ ਅਤੇ ਟੂਲਾਂ ਨਾਲ ਸੰConsistency ਢੰਗ ਨਾਲ ਜੁੜਨ ਲਈ AI ਏਜੰਟਾਂ ਨੂੰ ਸਮਰੱਥ ਬਣਾਉਂਦਾ ਹੈ।

ਆਓ MCP ਦੇ ਘਟਕਾਂ, ਸਿੱਧੇ API ਇਸਤੇਮਾਲ ਨਾਲ ਤੁਲਨਾ ਵਿੱਚ ਫਾਇਦੇ, ਅਤੇ ਇਹ ਦੇਖੀਏ ਕਿ AI ਏਜੰਟ ਕਿਸ ਤਰ੍ਹਾਂ MCP ਸਰਵਰ ਦੀ ਵਰਤੋਂ ਕਰ ਸਕਦੇ ਹਨ।

### MCP ਕੋਰ ਕੰਪੋਨੈਂਟ

MCP ਇੱਕ **ਕਲਾਇੰਟ-ਸਰਵਰ ਆਰਕੀਟੈਕਚਰ** 'ਤੇ ਕੰਮ ਕਰਦਾ ਹੈ ਅਤੇ ਮੁੱਖ ਕੰਪੋਨੈਂਟ ਇਹ ਹਨ:

• **Hosts** ਉਹ LLM ਐਪਲੀਕੇਸ਼ਨ ਹਨ (ਉਦਾਹਰਣ ਲਈ ਇੱਕ ਕੋਡ ਐਡੀਟਰ ਜਿਵੇਂ VSCode) ਜੋ MCP ਸਰਵਰ ਨਾਲ ਕਨੈਕਸ਼ਨ ਸ਼ੁਰੂ ਕਰਦੇ ਹਨ।

• **Clients** ਹੋਸਟ ਐਪਲੀਕੇਸ਼ਨ ਦੇ ਅੰਦਰ ਕੈੰਪੋਨੈਂਟ ਹਨ ਜੋ ਸਰਵਰਾਂ ਨਾਲ ਇਕ-ਤੋਂ-ਇਕ ਕਨੈਕਸ਼ਨਾਂ ਨੂੰ ਬਣਾਈ ਰੱਖਦੇ ਹਨ।

• **Servers** ਹਲਕੇ-ਫੁਲਕੇ ਪ੍ਰੋਗਰਾਮ ਹੁੰਦੇ ਹਨ ਜੋ ਨਿਰਧਾਰਤ ਸਮਰੱਥਾਵਾਂ ਉਪਲਬਧ ਕਰਵਾਉਂਦੇ ਹਨ।

ਪ੍ਰੋਟੋਕੋਲ ਵਿੱਚ ਤਿੰਨ ਮੁੱਖ ਪ੍ਰਿਮਿਟਿਵ ਸ਼ਾਮਲ ਹਨ ਜੋ ਇੱਕ MCP ਸਰਵਰ ਦੀਆਂ ਸਮਰੱਥਾਵਾਂ ਹਨ:

• **Tools**: ਇਹ ਵਿਭਾਜਿਤ ਕਾਰਵਾਈਆਂ ਜਾਂ ਫੰਕਸ਼ਨਾਂ ਹਨ ਜੋ ਇੱਕ AI ਏਜੰਟ ਕਾਰਜ ਕਰਨ ਲਈ ਕਾਲ ਕਰ ਸਕਦਾ ਹੈ। ਉਦਾਹਰਣ ਲਈ, ਇੱਕ ਮੌਸਮ ਸੇਵਾ "get weather" ਟੂਲ ਪ੍ਰਦਾਨ ਕਰ ਸਕਦੀ ਹੈ, ਜਾਂ ਇੱਕ ਈ-ਕਾਮਰਸ ਸਰਵਰ "purchase product" ਟੂਲ ਪ੍ਰਦਾਨ ਕਰ ਸਕਦਾ ਹੈ। MCP ਸਰਵਰ ਆਪਣੇ capability ਲਿਸਟਿੰਗ ਵਿੱਚ ਹਰ ਟੂਲ ਦਾ ਨਾਮ, ਵਰਣਨ ਅਤੇ ਇਨਪੁਟ/ਆਊਟਪੁਟ ਸਕੀਮਾ ਦਰਸਾਉਂਦੇ ਹਨ।

• **Resources**: ਇਹ ਪੜ੍ਹਨ-ਮਾਤਰ ਡੇਟਾ ਆਈਟਮ ਜਾਂ ਦਸਤਾਵੇਜ਼ ਹੁੰਦੇ ਹਨ ਜੋ MCP ਸਰਵਰ ਪ੍ਰਦਾਨ ਕਰ ਸਕਦਾ ਹੈ, ਅਤੇ ਕਲਾਇੰਟ ਉਨ੍ਹਾਂ ਨੂੰ ਮੰਗ 'ਤੇ ਪ੍ਰਾਪਤ ਕਰ ਸਕਦੇ ਹਨ। ਉਦਾਹਰਣਾਂ ਵਿੱਚ ਫਾਈਲ ਸਮੱਗਰੀ, ਡੇਟਾਬੇਸ ਰਿਕਾਰਡ, ਜਾਂ ਲੌਗ ਫਾਈਲਾਂ ਸ਼ਾਮਲ ਹਨ। Resources ਟੈਕਸਟ (ਜਿਵੇਂ ਕੋਡ ਜਾਂ JSON) ਜਾਂ ਬਾਇਨਰੀ (ਜਿਵੇਂ ਤਸਵੀਰਾਂ ਜਾਂ PDF) ਹੋ ਸਕਦੇ ਹਨ।

• **Prompts**: ਇਹ ਪਹਿਲਾਂ-ਤਿਆਰ ਟੈਂਪਲੇਟ ਹੁੰਦੇ ਹਨ ਜੋ ਸੁਝਾਏ ਗਏ ਪ੍ਰਾਪਟਾਂ ਪ੍ਰਦਾਨ ਕਰਦੇ ਹਨ, ਜਿਸ ਨਾਲ ਹੋਰ ਜਟਿਲ ਵਰਕਫਲੋਜ਼ ਬਣਾਉਣ ਵਿੱਚ ਸਹਾਇਤਾ ਮਿਲਦੀ ਹੈ।

### MCP ਦੇ ਫਾਇਦੇ

MCP AI ਏਜੰਟਾਂ ਲਈ ਮਹੱਤਵਪੂਰਨ ਲਾਭ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ:

• **ਡਾਇਨਾਮਿਕ ਟੂਲ ਖੋਜ**: ਏਜੰਟ ਇੱਕ ਸਰਵਰ ਤੋਂ ਉਪਲਬਧ ਟੂਲਾਂ ਦੀ ਸੂਚੀ ਗਤੀ ਸ਼ੀਲ ਤਰੀਕੇ ਨਾਲ ਪ੍ਰਾਪਤ ਕਰ ਸਕਦੇ ਹਨ, ਉਨ੍ਹਾਂ ਦੇ ਵਰਣਨ ਨਾਲ ਕਿ ਉਹ ਕੀ ਕਰਦੇ ਹਨ। ਇਹ ਰਵਾਇਤੀ APIs ਨਾਲ ਮੋਕਾਬਲੇ ਵਿੱਚ ਵੱਖਰਾ ਹੈ, ਜਿਹੜੇ ਅਕਸਰ ਇੰਟਿਗ੍ਰੇਸ਼ਨਾਂ ਲਈ ਸਥਿਰ ਕੋਡਿੰਗ ਦੀ ਲੋੜ ਰੱਖਦੇ ਹਨ, ਜਿਸਦਾ ਅਰਥ ਹੈ ਕਿ ਕਿਸੇ API ਵਿੱਚ ਬਦਲਾਅ ਲਈ ਕੋਡ ਅੱਪਡੇਟ ਦੀ ਜ਼ਰੂਰਤ ਹੁੰਦੀ ਹੈ। MCP "ਇੱਕ ਵਾਰੀ ਇੰਟਿਗ੍ਰੇਟ ਕਰੋ" ਵਿਧੀ ਪੇਸ਼ ਕਰਦਾ ਹੈ, ਜੋ ਵੱਧ ਅਨੁਕੂਲਤਾ ਦਿੰਦਾ ਹੈ।

• **ਵੱਖ-ਵੱਖ LLMs ਵਿੱਚ ਇੰਟਰਓਪਰੇਬਿਲਟੀ**: MCP ਵੱਖ-ਵੱਖ LLMs 'ਤੇ ਕੰਮ ਕਰਦਾ ਹੈ, ਇਹ ਲਚਕੀਲਾਪਨ ਦਿੰਦਾ ਹੈ ਤਾਂ ਜੋ ਕੋਰ ਮਾਡਲ ਨੂੰ ਬਿਹਤਰ ਪ੍ਰਦਰਸ਼ਨ ਲਈ ਬਦਲਾ ਜਾ ਸਕੇ।

• **ਮਿਆਰੀਕ੍ਰਿਤ ਸੁਰੱਖਿਆ**: MCP ਵਿੱਚ ਇੱਕ ਮਿਆਰੀਕ੍ਰਿਤ ਪ੍ਰਮਾਣੀਕਰਨ ਤਰੀਕਾ ਸ਼ਾਮਲ ਹੈ, ਜੋ ਵੱਖ-ਵੱਖ MCP ਸਰਵਰਾਂ ਤੱਕ ਪਹੁੰਚ ਜੋੜਨ ਵੇਲੇ ਸਕੇਲ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ। ਇਹ ਵਿਭਿੰਨ ਰਵਾਇਤੀ APIs ਲਈ ਵੱਖ-ਵੱਖ ਕੁੰਜੀਆਂ ਅਤੇ ਪ੍ਰਮਾਣਿਕਤਾ ਕਿਸਮਾਂ ਨੂੰ ਪ੍ਰਬੰਧਿਤ ਕਰਨ ਘਣਭੀੜਤੋਂ ਸਾਦਾ ਹੈ।

### MCP ਉਦਾਹਰਨ

![MCP ਡਾਇਗ੍ਰਾਮ](../../../translated_images/pa/mcp-diagram.e4ca1cbd551444a1.webp)

ਕਲਪਨਾ ਕਰੋ ਕਿ ਇੱਕ ਯੂਜ਼ਰ MCP ਪਾਵਰਡ AI ਸਹਾਇਕ ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਇੱਕ ਫਲਾਈਟ ਬੁੱਕ ਕਰਵਾਉਣਾ ਚਾਹੁੰਦਾ ਹੈ।

1. **ਕਨੈਕਸ਼ਨ**: AI ਸਹਾਇਕ (MCP ਕਲਾਇੰਟ) ਇੱਕ ਏਅਰਲਾਈਨ ਵਲੋਂ ਪ੍ਰਦਾਨ ਕੀਤੇ MCP ਸਰਵਰ ਨਾਲ ਕਨੈਕਟ ਕਰਦਾ ਹੈ।

2. **ਟੂਲ ਖੋਜ**: ਕਲਾਇੰਟ ਏਅਰਲਾਈਨ ਦੇ MCP ਸਰਵਰ ਤੋਂ ਪੁੱਛਦਾ ਹੈ, "ਤੁਹਾਡੇ ਕੋਲ ਕਿਹੜੇ ਟੂਲ ਉਪਲਬਧ ਹਨ?" ਸਰਵਰ "search flights" ਅਤੇ "book flights" ਵਰਗੇ ਟੂਲ ਨੂੰ ਜਵਾਬ ਵਜੋਂ ਭੇਜਦਾ ਹੈ।

3. **ਟੂਲ ਕਾਲ ਕਰਨਾ**: ਤੁਸੀਂ ਫਿਰ AI ਸਹਾਇਕ ਨੂੰ ਕਹਿੰਦੇ ਹੋ, "ਕਿਰਪਾ ਕਰਕੇ Portland ਤੋਂ Honolulu ਲਈ ਇੱਕ ਫਲਾਈਟ ਖੋਜੋ।" AI ਸਹਾਇਕ, ਆਪਣੇ LLM ਦੀ ਵਰਤੋਂ ਕਰਕੇ, ਪਛਾਣਦਾ ਹੈ ਕਿ ਇਸਨੂੰ "search flights" ਟੂਲ ਨੂੰ ਕਾਲ ਕਰਨ ਦੀ ਲੋੜ ਹੈ ਅਤੇ ਸਬੰਧਤ ਪੈਰਾਮੀਟਰ (origin, destination) MCP ਸਰਵਰ ਨੂੰ ਭੇਜਦਾ ਹੈ।

4. **ਕਾਰਜ ਅਤੇ ਜਵਾਬ**: MCP ਸਰਵਰ, ਇੱਕ ਰੈਪਰ ਵਜੋਂ ਕੰਮ ਕਰਦਿਆਂ, ਅਸਲ ਵਿੱਚ ਏਅਰਲਾਈਨ ਦੇ ਅੰਦਰੂਨੀ ਬੁਕਿੰਗ API ਨੂੰ ਕਾਲ ਕਰਦਾ ਹੈ। ਫਿਰ ਇਹ ਫਲਾਈਟ ਜਾਣਕਾਰੀ (ਉਦਾਹਰਣ ਲਈ JSON ਡੇਟਾ) ਪ੍ਰਾਪਤ ਕਰਕੇ AI ਸਹਾਇਕ ਨੂੰ ਵਾਪਸ ਭੇਜਦਾ ਹੈ।

5. **ਅੱਗੇ ਦੀ ਇੰਟਰੈਕਸ਼ਨ**: AI ਸਹਾਇਕ ਫਲਾਈਟ ਵਿਕਲਪ ਦਿਖਾਉਂਦਾ ਹੈ। ਜਦੋਂ ਤੁਸੀਂ ਇੱਕ ਫਲਾਈਟ ਚੁਣ ਲੈਂਦੇ ਹੋ, ਸਹਾਇਕ ਸ਼ਾਇਦ ਉਸੇ MCP ਸਰਵਰ 'ਤੇ "book flight" ਟੂਲ ਨੂੰ ਕਾਲ ਕਰਕੇ ਬੁਕਿੰਗ ਪੂਰੀ ਕਰ ਸਕਦਾ ਹੈ।

## ਏਜੰਟ-ਟੂ-ਏਜੰਟ ਪ੍ਰੋਟੋਕੋਲ (A2A)

ਜਿੱਥੇ MCP LLMs ਨੂੰ ਟੂਲਾਂ ਨਾਲ ਜੋੜਨ 'ਤੇ ਧਿਆਨ ਦਿੰਦਾ ਹੈ, ਓਥੇ **Agent-to-Agent (A2A) ਪ੍ਰੋਟੋਕੋਲ** ਇਕ ਕਦਮ ਅੱਗੇ ਵਧਦਾ ਹੈ ਅਤੇ ਵੱਖ-ਵੱਖ AI ਏਜੰਟਾਂ ਵਿਚਕਾਰ ਸੰਚਾਰ ਅਤੇ ਸਹਿਯੋਗ ਯੋਗ ਬਣਾਉਂਦਾ ਹੈ। A2A ਵੱਖ-ਵੱਖ ਸੰਗਠਨਾਂ, ਵਾਤਾਵਰਣਾਂ ਅਤੇ ਟੈਕ ਸਟੈਕਾਂ ਵਿੱਚੋਂ AI ਏਜੰਟਾਂ ਨੂੰ ਜੋੜਦਾ ਹੈ ਤਾਂ ਜੋ ਉਹ ਸਾਂਝੇ ਕਾਰਜ ਨੂੰ ਪੂਰਾ ਕਰ ਸਕਣ।

ਅਸੀਂ A2A ਦੇ ਘਟਕਾਂ ਅਤੇ ਫਾਇਦਿਆਂ ਦੀ ਜਾਂਚ ਕਰਾਂਗੇ, ਅਤੇ ਇਹ ਵੀ ਵੇਖਾਂਗੇ ਕਿ ਇਹ ਸਾਡੀ ਟ੍ਰੈਵਲ ਐਪਲੀਕੇਸ਼ਨ ਵਿੱਚ ਕਿਵੇਂ ਲਾਗੂ ਹੋ ਸਕਦਾ ਹੈ।

### A2A ਕੋਰ ਘਟਕ

A2A ਇਸ ਗੱਲ 'ਤੇ ਧਿਆਨ ਕੇਂਦਰਤ ਕਰਦਾ ਹੈ ਕਿ ਏਜੰਟਾਂ ਵਿਚਕਾਰ ਕਮਿਊਨੀਕੇਸ਼ਨ ਕਿਵੇਂ ਯੋਗ ਬਣਾਈ ਜਾਵੇ ਅਤੇ ਉਹ ਇਕੱਠੇ ਹੋ ਕੇ ਯੂਜ਼ਰ ਦੇ ਉਪ-ਕਾਰਜ ਨੂੰ ਪੂਰਾ ਕਰਨ। ਪ੍ਰੋਟੋਕੋਲ ਦੇ ਹਰ ਕੰਪੋਨੈਂਟ ਇਸ ਵਿੱਚ ਯੋਗਦਾਨ ਪਾਉਂਦੇ ਹਨ:

#### ਏਜੰਟ ਕਾਰਡ

ਇਸੇ ਤਰ੍ਹਾਂ ਜਿਵੇਂ MCP ਸਰਵਰ ਟੂਲਾਂ ਦੀ ਸੂਚੀ ਸਾਂਝੀ ਕਰਦਾ ਹੈ, ਇੱਕ Agent Card ਵਿੱਚ ਹੁੰਦਾ ਹੈ:
- ਏਜੰਟ ਦਾ ਨਾਮ।
- ਉਹਨਾਂ γενਰਲ ਟਾਸਕਾਂ ਦਾ **ਵਰਣਨ** ਜੋ ਇਹ ਪੂਰਾ ਕਰਦਾ ਹੈ।
- ਦੂਜੇ ਏਜੰਟਾਂ (ਜਾਂ ਇਨਸਾਨੀ ਉਪਭੋਗਤਿਆਂ) ਨੂੰ ਸਮਝਣ ਵਿੱਚ ਮਦਦ ਕਰਨ ਲਈ **ਨਿਰਧਾਰਤ ਹੁਨਰਾਂ ਦੀ ਸੂਚੀ** ਉਨ੍ਹਾਂ ਦੇ ਵਰਣਨਾਂ ਦੇ ਨਾਲ।
- ਏਜੰਟ ਦਾ **ਮੌਜੂਦਾ Endpoint URL**
- ਏਜੰਟ ਦਾ **ਵਰਜ਼ਨ** ਅਤੇ **ਸਮਰੱਥਾ** (ਉਦਾਹਰਣ ਲਈ streaming responses ਅਤੇ push notifications ਵਰਗੀਆਂ ਖ਼ਾਸੀਅਤਾਂ)।

#### ਏਜੰਟ ਐਕਜ਼ੈਕਿਊਟਰ

Agent Executor ਦੀ ਜ਼ਿੰਮੇਵਾਰੀ ਹੈ ਕਿ **ਯੂਜ਼ਰ ਚੈਟ ਦਾ ਸੰਦਰਭ ਰਿਮੋਟ ਏਜੰਟ ਤੱਕ ਪਹੁੰਚਾਇਆ ਜਾਵੇ**, ਕਿਉਂਕਿ ਰਿਮੋਟ ਏਜੰਟ ਨੂੰ ਸਮਝਣ ਲਈ ਇਹ ਲੋੜੀਂਦਾ ਹੈ ਕਿ ਕਿਹੜਾ ਕਾਰਜ ਪੂਰਾ ਕਰਨਾ ਹੈ। ਇੱਕ A2A ਸਰਵਰ ਵਿੱਚ, ਇੱਕ ਏਜੰਟ ਆਪਣੀ ਖੁਦੀ ਦੀ ਵੱਡੀ ਭਾਸ਼ਾ ਮਾਡਲ (LLM) ਦੀ ਵਰਤੋਂ ਕਰਦਾ ਹੈ ਆਉਂਦੇ ਹੋਏ ਬੇਨਤੀ-ਪਾਰਸ ਕਰਨ ਅਤੇ ਆਪਣੇ ਅੰਦਰੂਨੀ ਟੂਲਾਂ ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਕਾਰਜਾਂ ਨੂੰ ਨਿਭਾਉਣ ਲਈ।

#### ਆਰਟੀਫੈਕਟ

ਜਦੋਂ ਇੱਕ ਰਿਮੋਟ ਏਜੰਟ ਮੰਗੇ ਹੋਏ ਟਾਸਕ ਨੂੰ ਪੂਰਾ ਕਰ ਲੈਂਦਾ ਹੈ, ਤਾਂ ਉਸ ਦਾ ਕੰਮ ਦਾ ਉਤਪਾਦ ਇੱਕ ਆਰਟੀਫੈਕਟ ਵਜੋਂ ਬਣਾਇਆ ਜਾਂਦਾ ਹੈ। ਇੱਕ ਆਰਟੀਫੈਕਟ ਵਿੱਚ ਆਮ ਤੌਰ 'ਤੇ ਏਜੰਟ ਦੇ ਕੰਮ ਦਾ ਨਤੀਜਾ ਹੁੰਦਾ ਹੈ, ਕੀ ਪੂਰਾ ਕੀਤਾ ਗਿਆ ਉਸ ਦਾ ਵਰਣਨ, ਅਤੇ ਪਾਠ ਸੰਦਰਭ ਜੋ ਪ੍ਰੋਟੋਕੋਲ ਰਾਹੀਂ ਭੇਜਿਆ ਗਿਆ ਸੀ। ਆਰਟੀਫੈਕਟ ਭੇਜਣ ਤੋਂ ਬਾਅਦ, ਰਿਮੋਟ ਏਜੰਟ ਨਾਲ ਕਨੈਕਸ਼ਨ ਬੰਦ ਕਰ ਦਿੱਤੀ ਜਾਂਦੀ ਹੈ ਜਦ ਤੱਕ ਫਿਰ ਦੀ ਲੋੜ ਨਾ ਹੋਵੇ।

#### ਇਵੈਂਟ ਕਿਊ

ਇਹ ਕੰਪੋਨੈਂਟ ਅਪਡੇਟਸ ਅਤੇ ਸੁਨੇਹੇ ਭੇਜਣ ਲਈ ਵਰਤੀ ਜਾਂਦੀ ਹੈ। ਇਹ ਪ੍ਰੋਡਕਸ਼ਨ ਵਿੱਚ ਏਜੈਂਟਿਕ ਸਿਸਟਮਾਂ ਲਈ ਖ਼ਾਸ ਤੌਰ 'ਤੇ ਮਹੱਤਵਪੂਰਨ ਹੈ ਤਾਂ ਕਿ ਕੰਮ ਪੂਰਾ ਹੋਣ ਤੋਂ ਪਹਿਲਾਂ ਏਜੰਟਾਂ ਦੇ ਵਿਚਕਾਰ ਕਨੈਕਸ਼ਨ ਬੰਦ ਨਾ ਹੋ ਜਾਵੇ, ਖ਼ਾਸ ਕਰਕੇ ਜਦੋਂ ਟਾਸਕ ਨਿਪਟਾਰਾ ਲੰਮਾ ਸਮਾਂ ਲੈ ਸਕਦਾ ਹੈ।

### A2A ਦੇ ਫਾਇਦੇ

• **ਸੁਧਰੀ ਹੋਈ ਸਹਿਯੋਗਤਾ**: ਇਹ ਵੱਖ-ਵੱਖ ਵੇਂਡਰਾਂ ਅਤੇ ਪਲੈਟਫਾਰਮਾਂ ਦੇ ਏਜੰਟਾਂ ਨੂੰ ਇੰਟਰੈਕਟ ਅਤੇ ਸੰਦਰਭ ਸਾਂਝਾ ਕਰਨ ਯੋਗ ਬਣਾਉਂਦਾ ਹੈ, ਤੇ ਪਰੰਪਰਾਗਤ ਤੌਰ 'ਤੇ ਵੱਖਰੇ ਸਿਸਟਮਾਂ ਦੇ ਦਰਮਿਆਨ ਸਹਿਯੋਗ ਨੂੰ ਆਸਾਨ ਬਣਾਉਂਦਾ ਹੈ।

• **ਮਾਡਲ ਚੋਣ ਵਿੱਚ ਲਚਕੀਲਾਪਨ**: ਹਰ A2A ਏਜੰਟ ਆਪਣੇ ਬੇਨਤੀ ਦੀ ਸੇਵਾ ਕਰਨ ਲਈ ਆਪਣਾ LLM ਚੁਣ ਸਕਦਾ ਹੈ, ਜਿਸ ਨਾਲ ਹਰ ਏਜੰਟ ਲਈ ਅਪਟੀਮਾਈਜ਼ਡ ਜਾਂ ਫਾਈਨ-ਟਿਊਨ ਕੀਤਾ ਮਾਡਲ ਵਰਤਣ ਦੀ ਆਜ਼ਾਦੀ ਮਿਲਦੀ ਹੈ, ਜਿਵੇਂ ਕਿ ਕੁਝ MCP ਸੰਦਰਭਾਂ ਵਿੱਚ ਇੱਕ ਇਕੱਲੇ LLM ਕਨੈਕਸ਼ਨ ਹੋ ਸਕਦਾ ਹੈ।

• **ਬਿਲਟ-ਇਨ ਪ੍ਰਮਾਣੀਕਰਨ**: ਪ੍ਰਮਾਣੀਕਰਨ ਸਿੱਧਾ A2A ਪ੍ਰੋਟੋਕੋਲ ਵਿੱਚ ਸ਼ਾਮਲ ਹੁੰਦਾ ਹੈ, ਜੋ ਏਜੰਟ ਇੰਟਰੈਕਸ਼ਨਾਂ ਲਈ ਮਜ਼ਬੂਤ ਸੁਰੱਖਿਆ ਫਰੇਮਵਰਕ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ।

### A2A ਉਦਾਹਰਨ

![A2A ਡਾਇਗ੍ਰਾਮ](../../../translated_images/pa/A2A-Diagram.8666928d648acc26.webp)

ਆਓ ਆਪਣੀ ਟ੍ਰੈਵਲ ਬੁਕਿੰਗ ਸਥਿਤੀ ਨੂੰ ਵਧਾਈਏ, ਪਰ ਇਸ ਵਾਰੀ A2A ਦੀ ਵਰਤੋਂ ਕਰ ਕੇ।

1. **ਬਹੁ-ਏਜੰਟ ਲਈ ਯੂਜ਼ਰ ਬੇਨਤੀ**: ਇੱਕ ਯੂਜ਼ਰ "ਟ੍ਰੈਵਲ ਏਜੰਟ" A2A ਕਲਾਇੰਟ/ਏਜੰਟ ਨਾਲ ਬਾਤ ਕਰਦਾ ਹੈ, ਉਦਾਹਰਣ ਲਈ: "ਕਿਰਪਾ ਕਰਕੇ ਅਗਲੇ ਹਫਤੇ ਹੋਨੋਲੂਲੁ ਲਈ ਪੂਰਾ ਟ੍ਰਿਪ ਬੁੱਕ ਕਰੋ, ਜਿਸ ਵਿੱਚ ਫਲਾਈਟ, ਹੋਟਲ ਅਤੇ ਰੈਂਟਲ ਕਾਰ ਸ਼ਾਮਲ ਹੋਣ।"

2. **ਟ੍ਰੈਵਲ ਏਜੰਟ ਦੁਆਰਾ ਅਰਕੈਸਟ੍ਰੇਸ਼ਨ**: ਟ੍ਰੈਵਲ ਏਜੰਟ ਇਸ ਜਟਿਲ ਬੇਨਤੀ ਨੂੰ ਪ੍ਰਾਪਤ ਕਰਦਾ ਹੈ। ਆਪਣੀ LLM ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਇਹ ਤਰਕ ਕਰਦਾ ਹੈ ਅਤੇ ਫੈਸਲਾ ਕਰਦਾ ਹੈ ਕਿ ਇਹਨੂੰ ਹੋਰ ਵਿਸ਼ੇਸ਼ ਏਜੰਟਾਂ ਨਾਲ ਇੰਟਰੈਕਟ ਕਰਨ ਦੀ ਲੋੜ ਹੈ।

3. **ਏਜੰਟ-ਦਰ-ਏਜੰਟ ਸੰਚਾਰ**: ਟ੍ਰੈਵਲ ਏਜੰਟ ਫਿਰ A2A ਪ੍ਰੋਟੋਕੋਲ ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਡਾਊਨਸਟਰੀਮ ਏਜੰਟਾਂ ਨਾਲ ਕਨੈਕਟ ਕਰਦਾ ਹੈ, ਜਿਵੇਂ "Airline Agent", "Hotel Agent", ਅਤੇ "Car Rental Agent" ਜੋ ਵੱਖ-ਵੱਖ ਕੰਪਨੀਆਂ ਦੁਆਰਾ ਬਣਾਏ ਗਏ ਹੋ ਸਕਦੇ ਹਨ।

4. **ਡੈਲੇਗੇਟ ਕੀਤਾ ਟਾਸਕ ਨਿਪਟਾਰਾ**: ਟ੍ਰੈਵਲ ਏਜੰਟ ਇਹਨਾਂ ਵਿਸ਼ੇਸ਼ ਏਜੰਟਾਂ ਨੂੰ ਨਿਰਧਾਰਤ ਕਾਰਜ ਭੇਜਦਾ ਹੈ (ਜਿਵੇਂ, "Honolulu ਲਈ ਫਲਾਈਟ ਲੱਭੋ", "ਹੋਟਲ ਬੁੱਕ ਕਰੋ", "ਕਾਰ ਰੈਂਟ ਤੇ ਲਓ")। ਹਰ ਇਕ ਵਿਸ਼ੇਸ਼ ਏਜੰਟ, ਆਪਣੀ LLM ਚਲਾਉਂਦਾ ਹੈ ਅਤੇ ਆਪਣੇ ਟੂਲਾਂ ਦੀ ਵਰਤੋਂ ਕਰਦਾ ਹੈ (ਜੋ khud MCP ਸਰਵਰ ਹੋ ਸਕਦੇ ਹਨ), ਅਤੇ ਬੁਕਿੰਗ ਦਾ ਉਹ ਖਾਸ ਹਿੱਸਾ ਨਿਭਾਂਦਾ ਹੈ।

5. **ਸੰਯੁਕਤ ਜਵਾਬ**: ਜਦੋਂ ਸਾਰੇ ਡਾਊਨਸਟਰੀਮ ਏਜੰਟ ਆਪਣਾ ਕੰਮ ਪੂਰਾ ਕਰ ਲੈਂਦੇ ਹਨ, ਟ੍ਰੈਵਲ ਏਜੰਟ ਨਤੀਜਿਆਂ (ਫਲਾਈਟ ਵੇਰਵੇ, ਹੋਟਲ ਪੁਸ਼ਟੀਕਰਨ, ਕਾਰ ਰੈਂਟਲ ਬੁਕਿੰਗ) ਨੂੰ ਇਕੱਠਾ ਕਰਕੇ ਯੂਜ਼ਰ ਨੂੰ ਇੱਕ ਸੰਪੂਰਨ, ਚੈਟ-ਸ਼ੈਲੀ ਜਵਾਬ ਭੇਜਦਾ ਹੈ।

## ਕੁਦਰਤੀ ਭਾਸ਼ਾ ਵੈੱਬ (NLWeb)

ਵੈੱਬਸਾਈਟ ਲੰਬੇ ਸਮੇਂ ਤੋਂ ਉਪਭੋਗਤਿਆਂ ਲਈ ਇੰਟਰਨੈੱਟ 'ਤੇ ਜਾਣਕਾਰੀ ਅਤੇ ਡੇਟਾ ਪ੍ਰਾਪਤ ਕਰਨ ਦਾ ਪ੍ਰਮੁੱਖ ਤਰੀਕਾ ਰਹੀ ਹੈ।

ਆਓ NLWeb ਦੇ ਵੱਖ-ਵੱਖ ਘਟਕ, NLWeb ਦੇ ਫਾਇਦੇ ਅਤੇ ਸਾਡੀ ਟ੍ਰੈਵਲ ਐਪਲੀਕੇਸ਼ਨ ਵੱਲੋਂ NLWeb ਕਿਵੇਂ ਕੰਮ ਕਰਦਾ ਹੈ, ਦੇਖੀਏ।

### NLWeb ਦੇ ਘਟਕ

- **NLWeb Application (Core Service Code)**: ਉਹ ਸਿਸਟਮ ਜੋ ਕੁਦਰਤੀ ਭਾਸ਼ਾ ਦੇ ਸਵਾਲਾਂ ਨੂੰ ਪ੍ਰੋਸੈਸ ਕਰਦਾ ਹੈ। ਇਹ ਪਲੇਟਫਾਰਮ ਦੇ ਵੱਖ-ਵੱਖ ਭਾਗਾਂ ਨੂੰ ਜੋੜ ਕੇ ਜਵਾਬ ਬਣਾਉਂਦਾ ਹੈ। ਤੁਸੀਂ ਇਸਨੂੰ ਵੈੱਬਸਾਈਟ ਦੀਆਂ ਕੁਦਰਤੀ ਭਾਸ਼ਾ ਵਿਸ਼ੇਸ਼ਤਾਵਾਂ ਚਲਾਉਣ ਵਾਲੇ **ਇੰਜਨ** ਵਜੋਂ ਸੋਚ ਸਕਦੇ ਹੋ।

- **NLWeb Protocol**: ਇਹ ਵੈੱਬਸਾਈਟ ਨਾਲ ਕੁਦਰਤੀ ਭਾਸ਼ਾ ਇੰਟਰੈਕਸ਼ਨ ਲਈ ਨਿਯਮਾਂ ਦਾ ਇੱਕ **ਮੂਲ ਸੈੱਟ** ਹੈ। ਇਹ JSON ਫਾਰਮੈਟ ਵਿੱਚ ਜਵਾਬ ਵਾਪਸ ਭੇਜਦਾ ਹੈ (ਅਕਸਰ Schema.org ਦੀ ਵਰਤੋਂ ਕਰਕੇ)। ਇਸਦਾ ਮਕਸਦ "AI ਵੈੱਬ" ਲਈ ਇੱਕ ਸਧਾਰਨ ਬੁਨਿਆਦ ਬਣਾਉਣਾ ਹੈ, ਉਸੇ ਤਰੀਕੇ ਨਾਲ ਜਿਸ ਤਰ੍ਹਾਂ HTML ਨੇ ਆਨਲਾਈਨ ਦਸਤਾਵੇਜ਼ ਸਾਂਝੇ ਕਰਨ ਸਰਲ ਬਣਾਏ।

- **MCP Server (Model Context Protocol Endpoint)**: ਹਰ NLWeb ਸੈਟਅਪ ਇੱਕ **MCP ਸਰਵਰ** ਵਜੋਂ ਵੀ ਕੰਮ ਕਰਦਾ ਹੈ। ਇਸਦਾ ਅਰਥ ਇਹ ਹੈ ਕਿ ਇਹ ਹੋਰ AI ਸਿਸਟਮਾਂ ਨਾਲ **ਟੂਲ (ਜਿਵੇਂ ਇੱਕ “ask” ਮੈਥਡ) ਅਤੇ ਡੇਟਾ** ਸਾਂਝਾ ਕਰ ਸਕਦਾ ਹੈ। ਅਮਲ ਵਿੱਚ, ਇਹ ਵੈੱਬਸਾਈਟ ਦੀ ਸਮੱਗਰੀ ਅਤੇ ਸਮਰੱਥਾਵਾਂ ਨੂੰ AI ਏਜੰਟਾਂ ਲਈ ਯੂਜ਼ੇਬਲ ਬਣਾਉਂਦਾ ਹੈ, ਜਿਸ ਨਾਲ ਸਾਈਟ ਵੱਡੇ "ਏਜੰਟ ਇਕੋਸਿਸਟਮ" ਦਾ ਹਿੱਸਾ ਬਣ ਸਕਦੀ ਹੈ।

- **Embedding Models**: ਇਹ ਮਾਡਲ ਵੈੱਬਸਾਈਟ ਸਮੱਗਰੀ ਨੂੰ ਨੰਬਰਿਕ ਪ੍ਰਤੀਨਿਧੀਆਂ (vectors) ਵਿਚ ਬਦਲਣ ਲਈ ਵਰਤੇ ਜਾਂਦੇ ਹਨ (ਜਿਨ੍ਹਾਂ ਨੂੰ embeddings ਕਿਹਾ ਜਾਂਦਾ ਹੈ)। ਇਹ vectors ਅਰਥ ਨੂੰ ਕੈਪਚਰ ਕਰਦੇ ਹਨ ਜਿਸ ਨੂੰ ਕੰਪਿਊਟਰ ਤੁਲਨਾ ਅਤੇ ਖੋਜ ਲਈ ਵਰਤ ਸਕਦੇ ਹਨ। ਇਹ ਇੱਕ ਵਿਸ਼ੇਸ਼ ਡੇਟਾਬੇਸ ਵਿੱਚ ਸੰਭਾਲੇ ਜਾਂਦੇ ਹਨ, ਅਤੇ ਉਪਭੋਗਤਾ ਚੁਣ ਸਕਦੇ ਹਨ ਕਿ ਉਹ ਕਿਹੜਾ embedding ਮਾਡਲ ਵਰਤਣਾ ਚਾਹੁੰਦੇ ਹਨ।

- **Vector Database (Retrieval Mechanism)**: ਇਹ ਡੇਟਾਬੇਸ ਵੈੱਬਸਾਈਟ ਸਮੱਗਰੀ ਦੇ embeddings ਨੂੰ ਸਟੋਰ ਕਰਦਾ ਹੈ। ਜਦੋਂ ਕੋਈ ਸਵਾਲ ਪੁੱਛਿਆ ਜਾਂਦਾ ਹੈ, NLWeb ਤੇਜ਼ੀ ਨਾਲ ਸਭ ਤੋਂ ਸਬੰਧਤ ਜਾਣਕਾਰੀ ਲੱਭਣ ਲਈ vector database ਨੂੰ ਚੈਕ ਕਰਦਾ ਹੈ। ਇਹ ਸਮਰੱਥਾ-ਅਧਾਰਿਤ ਸੰਭਾਵਿਤ ਜਵਾਬਾਂ ਦੀ ਸੂਚੀ ਦੇਂਦਾ ਹੈ। NLWeb Qdrant, Snowflake, Milvus, Azure AI Search, ਅਤੇ Elasticsearch ਵਰਗੇ ਵੱਖ-ਵੱਖ ਵੈਕਟਰ ਸਟੋਰੇਜ ਸਿਸਟਮਾਂ ਨਾਲ ਕੰਮ ਕਰਦਾ ਹੈ।

### NLWeb ਉਦਾਹਰਨ

![NLWeb](../../../translated_images/pa/nlweb-diagram.c1e2390b310e5fe4.webp)

ਆਓ ਸਾਡੀ ਟ੍ਰੈਵਲ ਬੁਕਿੰਗ ਵੈੱਬਸਾਈਟ ਨੂੰ ਫਿਰ ਸੋਚੀਏ, ਪਰ ਇਸ ਵਾਰੀ ਇਹ NLWeb ਨਾਲ ਪਾਵਰਡ ਹੈ।

1. **ਡੇਟਾ ਇੰਗੇਸ਼ਨ**: ਟ੍ਰੈਵਲ ਵੈੱਬਸਾਈਟ ਦੇ ਮੌਜੂਦਾ ਪਹੁੰਚ ਉਤਪਾਦ ਕੈਟਾਲੋਗ (ਉਦਾਹਰਣ ਲਈ ਫਲਾਈਟ ਲਿਸਟਿੰਗ, ਹੋਟਲ ਵਰਣਨ, ਟੂਰ ਪੈਕੇਜ) Schema.org ਦੇ ਫਾਰਮੈਟ ਵਿੱਚ ਜਾਂ RSS ਫੀਡਾਂ ਰਾਹੀਂ ਲੋਡ ਕੀਤੇ ਜਾਂਦੇ ਹਨ। NLWeb ਦੇ ਟੂਲ ਇਹ ਸਟ੍ਰੱਕਚਰਡ ਡੇਟਾ ਇੰਗੇਸਟ ਕਰਦੇ ਹਨ, embeddings ਬਣਾਉਂਦੇ ਹਨ, ਅਤੇ ਉਨ੍ਹਾਂ ਨੂੰ ਲੋਕਲ ਜਾਂ ਰਿਮੋਟ vector ਡੇਟਾਬੇਸ ਵਿੱਚ ਸਟੋਰ ਕਰਦੇ ਹਨ।

2. **ਕੁਦਰਤੀ ਭਾਸ਼ਾ ਪ੍ਰਸ਼ਨ (ਇਨਸਾਨ)**: ਇੱਕ ਯੂਜ਼ਰ ਵੈੱਬਸਾਈਟ 'ਤੇ ਆਉਂਦਾ ਹੈ ਅਤੇ ਮੈਨੂਜ਼ ਸੰਭਾਲਣ ਦੀ ਥਾਂ ਇੱਕ ਚੈਟ ਇੰਟਰਫੇਸ ਵਿੱਚ ਲਿਖਦਾ ਹੈ: "ਮੇਰੇ ਲਈ ਅਗਲੇ ਹਫਤੇ ਹੋਨੋਲੂਲੁ ਵਿੱਚ ਪਰਿਵਾਰ-ਅਨੁਕੂਲ ਹੋਟਲ ਲੱਭੋ ਜਿਸ ਵਿੱਚ ਪੂਲ ਹੋਵੇ।"

3. **NLWeb ਪ੍ਰੋਸੈਸਿੰਗ**: NLWeb ਐਪਲੀਕੇਸ਼ਨ ਇਹ ਪ੍ਰਸ਼ਨ ਪ੍ਰਾਪਤ ਕਰਦਾ ਹੈ। ਇਹ ਸਮਝਣ ਲਈ ਪ੍ਰਸ਼ਨ ਨੂੰ ਇੱਕ LLM ਨੂੰ ਭੇਜਦਾ ਹੈ ਅਤੇ ਸਾਂਝੇ ਸਮੇਂ ਵਿੱਚ ਆਪਣੇ vector ਡੇਟਾਬੇਸ ਨੂੰ ਸਬੰਧਤ ਹੋਟਲ ਲਿਸਟਿੰਗ ਲਈ ਖੋਜਦਾ ਹੈ।

4. **ਸਹੀ ਨਤੀਜੇ**: LLM ਡੇਟਾਬੇਸ ਤੋਂ ਖੋਜ ਨਤੀਜਿਆਂ ਦੀ ਵਿਆਖਿਆ ਕਰਨ ਵਿੱਚ ਮਦਦ ਕਰਦਾ ਹੈ, "ਪਰਿਵਾਰ-ਅਨੁਕੂਲ", "ਪੂਲ", ਅਤੇ "ਹੋਨੋਲੂਲੁ" ਮਾਪਦੰਡਾਂ ਦੇ ਆਧਾਰ 'ਤੇ ਸਭ ਤੋਂ ਵਧੀਆ ਮਿਲਾਉਂ ਨੂੰ ਪਛਾਣਦਾ ਹੈ, ਅਤੇ ਫਿਰ ਇੱਕ ਕੁਦਰਤੀ ਭਾਸ਼ਾ ਜਵਾਬ ਫਾਰਮੈਟ ਕਰਦਾ ਹੈ। ਬਹੁਤ ਜ਼ਰੂਰੀGall: ਜਵਾਬ ਵੈੱਬਸਾਈਟ ਦੇ ਕੈਟਾਲੋਗ ਵਿੱਚੋਂ ਅਸਲੀ ਹੋਟਲਾਂ ਨੂੰ ਦਰਸਾਉਂਦਾ ਹੈ, ਕਿਉਂਕਿ ਨਿਰਾਖਤ ਜਾਣਕਾਰੀ ਤੋਂ ਬਚਣਾ ਮਹੱਤਵਪੂਰਨ ਹੈ।

5. **AI ਏਜੰਟ ਇੰਟਰੈਕਸ਼ਨ**: ਕਿਉਂਕਿ NLWeb ਇੱਕ MCP ਸਰਵਰ ਵਜੋਂ ਕੰਮ ਕਰਦਾ ਹੈ, ਬਾਹਰੀ AI ਟ੍ਰੈਵਲ ਏਜੰਟ ਵੀ ਇਸ ਵੈੱਬਸਾਈਟ ਦੀ NLWeb ਇੰਸਟੈਂਸ ਨਾਲ ਕਨੈਕਟ ਕਰ ਸਕਦਾ ਹੈ। AI ਏਜੰਟ ਫਿਰ `ask("Are there any vegan-friendly restaurants in the Honolulu area recommended by the hotel?")` MCP ਮੈਥਡ ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਸਾਈਟ ਨੂੰ ਸਿੱਧਾ ਪ੍ਰਸ਼ਨ ਪੁੱਛ ਸਕਦਾ ਹੈ। NLWeb ਇੰਸਟੈਂਸ ਇਸ ਨੂੰ ਪ੍ਰੋਸੈਸ ਕਰੇਗਾ, ਆਪਣੀ ਰੈਸਟੋਰੈਂਟ ਜਾਣਕਾਰੀ ਦੀ ਡੇਟਾਬੇਸ (ਜੇ ਲੋਡ ਕੀਤੀ ਹੋਵੇ) ਦੀ ਵਰਤੋਂ ਕਰਕੇ, ਅਤੇ ਇੱਕ ਸੰਰਚਿਤ JSON ਜਵਾਬ ਮੁਹੱਈਆ ਕਰਵੇਗਾ।
