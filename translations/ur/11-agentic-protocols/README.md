# ایجنٹک پروٹوکولز کا استعمال (MCP، A2A اور NLWeb)

[![ایجنٹک پروٹوکولز](../../../translated_images/ur/lesson-11-thumbnail.b6c742949cf1ce2a.webp)](https://youtu.be/X-Dh9R3Opn8)

> _(ویڈیو دیکھنے کے لیے اوپر دی گئی تصویر پر کلک کریں)_

جیسا کہ AI ایجنٹس کا استعمال بڑھ رہا ہے، اسی طرح ایسے پروٹوکولز کی ضرورت بھی بڑھ رہی ہے جو معیاری، محفوظ، اور کھلی جدت کو یقینی بنائیں۔ اس سبق میں، ہم 3 پروٹوکولز کا احاطہ کریں گے جو اس ضرورت کو پورا کرنے کی کوشش کرتے ہیں - ماڈل کانٹیکسٹ پروٹوکول (MCP)، ایجنٹ ٹو ایجنٹ (A2A) اور نیچرل لینگویج ویب (NLWeb)۔

## تعارف

اس سبق میں ہم شامل کریں گے:

• کیسے **MCP** AI ایجنٹس کو بیرونی ٹولز اور ڈیٹا تک رسائی دیتا ہے تاکہ صارف کے کام مکمل کیے جا سکیں۔

• کیسے **A2A** مختلف AI ایجنٹس کے درمیان مواصلت اور تعاون ممکن بناتا ہے۔

• کیسے **NLWeb** کسی بھی ویب سائٹ پر قدرتی زبان کے انٹرفیس فراہم کرتا ہے، جس سے AI ایجنٹس ویب مواد کو دریافت اور تعامل کر سکتے ہیں۔

## تعلیمی اہداف

• AI ایجنٹس کے تناظر میں MCP، A2A، اور NLWeb کے بنیادی مقصد اور فوائد کی **شناخت** کریں۔

• سمجھائیں کہ ہر پروٹوکول بڑے لینگویج ماڈلز، ٹولز، اور دیگر ایجنٹس کے درمیان مواصلت اور تعامل کس طرح آسان بناتا ہے۔

• پہچانیں کہ ہر پروٹوکول پیچیدہ ایجنٹک سسٹمز کی تعمیر میں کون سا مختلف کردار ادا کرتا ہے۔

## ماڈل کانٹیکسٹ پروٹوکول

**ماڈل کانٹیکسٹ پروٹوکول (MCP)** ایک اوپن اسٹینڈرڈ ہے جو ایپلیکیشنز کو LLMs کو کانٹیکسٹ اور ٹولز فراہم کرنے کا معیاری طریقہ فراہم کرتا ہے۔ یہ ایک "یونیورسل ایڈاپٹر" مہیا کرتا ہے جو مختلف ڈیٹا ذرائع اور ٹولز سے AI ایجنٹس کو مستقل اور متحد انداز میں جوڑتا ہے۔

آئیں MCP کے اجزاء، براہِ راست API استعمال سے اس کے فوائد اور ایک مثال دیکھیں کہ AI ایجنٹس MCP سرور کو کس طرح استعمال کر سکتے ہیں۔

### MCP کے بنیادی اجزاء

MCP ایک **کلائنٹ-سرور آرکیٹیکچر** پر کام کرتا ہے اور اس کے بنیادی اجزاء درج ذیل ہیں:

• **میزبان (Hosts)** وہ LLM ایپلیکیشنز ہیں (جیسا کہ VSCode کوڈ ایڈیٹر) جو MCP سرور کے ساتھ کنکشن شروع کرتے ہیں۔

• **کلائنٹس (Clients)** میزبان ایپ کے اندر وہ جزو ہیں جو سرورز کے ساتھ ایک سے ایک کنکشن برقرار رکھتے ہیں۔

• **سرورز (Servers)** ہلکے پھلکے پروگرام ہیں جو مخصوص صلاحیتوں کو فراہم کرتے ہیں۔

پروٹوکول میں تین بنیادی پرمیٹس شامل ہیں جو MCP سرور کی صلاحیتیں ہیں:

• **ٹولز**: یہ ایسی مخصوص کارروائیاں یا فنکشنز ہیں جو AI ایجنٹ کسی عمل کو کرنے کے لیے کال کر سکتا ہے۔ مثال کے طور پر، ایک موسم کی سروس "موسم معلوم کریں" کا ٹول پیش کر سکتی ہے، یا ای کامرس سرور "مصنوعات خریدنا" کا ٹول فراہم کر سکتا ہے۔ MCP سرور ہر ٹول کا نام، تفصیل اور ان پٹ/آؤٹ پٹ اسکیمہ اپنی صلاحیتوں کی فہرست میں پیش کرتا ہے۔

• **ذرائع (Resources)**: یہ وہ صرف پڑھنے کے قابل ڈیٹا آئٹمز یا دستاویزات ہیں جو MCP سرور فراہم کر سکتا ہے، اور کلائنٹس ضرورت پر انہیں حاصل کر سکتے ہیں۔ مثالیں فائل کے مواد، ڈیٹا بیس ریکارڈز، یا لاگ فائلز شامل ہیں۔ ذرائع متن (جیسا کہ کوڈ یا JSON) یا بائنری (جیسا کہ تصاویر یا PDFs) ہو سکتے ہیں۔

• **پرومپٹس (Prompts)**: یہ پہلے سے طے شدہ ٹیمپلیٹس ہیں جو مجوزہ پرومپٹس فراہم کرتے ہیں، جس سے زیادہ پیچیدہ ورک فلو ممکن ہوتے ہیں۔

### MCP کے فوائد

MCP AI ایجنٹس کے لیے اہم فوائد پیش کرتا ہے:

• **ڈائنامک ٹول دریافت**: ایجنٹس سرور سے دستیاب ٹولز کی فہرست اور ان کے کاموں کی تفصیل موصول کر سکتے ہیں۔ یہ روایتی APIs سے مختلف ہے جہاں انضمام کے لیے اکثر مستحکم کوڈنگ درکار ہوتی ہے، یعنی ہر API تبدیلی پر کوڈ اپ ڈیٹ کرنا پڑتا ہے۔ MCP "ایک بار انضمام" کا طریقہ پیش کرتا ہے، جو زیادہ لچکدار ہے۔

• **مختلف LLMs کے ساتھ انٹرآپریبیلٹی**: MCP مختلف LLMs پر کام کرتا ہے، جس سے بہتر کارکردگی کے لیے کور ماڈلز تبدیل کرنا آسان ہوتا ہے۔

• **معیاری سیکیورٹی**: MCP ایک معیاری تصدیقی طریقہ شامل کرتا ہے، جس سے مزید MCP سرورز تک رسائی کا اضافہ آسان ہو جاتا ہے۔ یہ روایتی APIs کے مختلف keys اور تصدیقی اقسام کو سنبھالنے سے آسان ہے۔

### MCP کی مثال

![MCP Diagram](../../../translated_images/ur/mcp-diagram.e4ca1cbd551444a1.webp)

فرض کریں کہ صارف MCP سے چلنے والے AI اسسٹنٹ کا استعمال کرتے ہوئے پرواز بک کرنا چاہتا ہے۔

1. **کنکشن**: AI اسسٹنٹ (MCP کلائنٹ) ایئرلائن کے فراہم کردہ MCP سرور سے جڑتا ہے۔

2. **ٹول دریافت**: کلائنٹ ایئرلائن کے MCP سرور سے پوچھتا ہے، "آپ کے پاس کون سے ٹولز دستیاب ہیں؟" سرور "search flights" اور "book flights" جیسے ٹولز کا جواب دیتا ہے۔

3. **ٹول کال**: پھر آپ AI اسسٹنٹ سے کہتے ہیں، "پلیز پورٹ لینڈ سے ہونولولو کے لیے فلائٹ تلاش کریں۔" AI اسسٹنٹ، اپنے LLM کا استعمال کرتے ہوئے، سمجھتا ہے کہ اسے "search flights" ٹول کال کرنا ہے اور متعلقہ پیرامیٹرز (ماخذ، منزل) MCP سرور کو بھیجتا ہے۔

4. **عمل درآمد اور جواب**: MCP سرور، جو ایک ریپر کے طور پر کام کرتا ہے، ایئرلائن کے اندرونی بکنگ API کو کال کرتا ہے۔ پھر اسے پرواز کی معلومات (مثلاً JSON ڈیٹا) موصول ہوتی ہے اور AI اسسٹنٹ کو بھیج دی جاتی ہے۔

5. **مزید تعامل**: AI اسسٹنٹ پرواز کے اختیارات دکھاتا ہے۔ جب آپ پرواز منتخب کرتے ہیں، تو اسسٹنٹ ممکنہ طور پر "book flight" ٹول کو اسی MCP سرور پر کال کر کے بکنگ مکمل کرتا ہے۔

## ایجنٹ ٹو ایجنٹ پروٹوکول (A2A)

جہاں MCP LLMs کو ٹولز سے منسلک کرنے پر توجہ دیتا ہے، **ایجنٹ ٹو ایجنٹ (A2A) پروٹوکول** ایک قدم آگے بڑھ کر مختلف AI ایجنٹس کے درمیان مواصلت اور تعاون کو ممکن بناتا ہے۔ A2A مختلف تنظیموں، ماحولیات، اور ٹیکنالوجی اسٹیکس کے AI ایجنٹس کو مشترکہ کام مکمل کرنے کے لیے جوڑتا ہے۔

ہم A2A کے اجزاء اور فوائد کا جائزہ لیں گے اور ہمارے سفر کے ایپلیکیشن کی مثال دیں گے کہ یہ کیسے کام کر سکتا ہے۔

### A2A کے بنیادی اجزاء

A2A ایجنٹس کے درمیان بات چیت کو فعال بنانے اور انہیں صارف کے ذیلی کام کو مکمل کرنے کے لیے مل کر کام کرنے پر مرکوز ہے۔ پروٹوکول کے ہر جزو کا اس میں کردار ہے:

#### ایجنٹ کارڈ

جیسے MCP سرور ٹولز کی فہرست شیئر کرتا ہے، ایجنٹ کارڈ میں شامل ہوتا ہے:  
- ایجنٹ کا نام۔  
- اس کے عمومی کاموں کی **تفصیل**۔  
- مخصوص مہارتوں کی فہرست اور وہ تفصیل جو دوسرے ایجنٹس (یا انسان صارفین) کو سمجھنے میں مدد دیتی ہیں کہ کب اور کیوں وہ ایجنٹ کو کال کریں۔  
- ایجنٹ کا موجودہ اینڈ پوائنٹ URL  
- ایجنٹ کا **ورژن** اور **صلاحیتیں**، جیسا کہ اسٹریمنگ ریسپانسز اور پش نوٹیفیکیشنز۔

#### ایجنٹ ایگزیکیٹر

ایجنٹ ایگزیکیٹر کی ذمہ داری ہے کہ **صارف کے چیٹ کے کانٹیکسٹ کو ریموٹ ایجنٹ تک منتقل کرے**، تاکہ ریموٹ ایجنٹ اس کام کو سمجھ سکے جو مکمل کرنا ہے۔ A2A سرور میں، ایجنٹ اپنے LLM کا استعمال کرتا ہے تاکہ آنے والی درخواستوں کو پارس کرے اور اپنے داخلی ٹولز کے ذریعے کام انجام دے۔

#### آرٹیفیکٹ

جب ریموٹ ایجنٹ مطلوبہ کام مکمل کر لیتا ہے، تو اس کا نتیجہ آرٹیفیکٹ کی صورت میں بنایا جاتا ہے۔ آرٹیفیکٹ میں **ایجنٹ کے کام کا نتیجہ، مکمل کی گئی چیز کی تفصیل،** اور وہ **متنی کانٹیکسٹ** ہوتا ہے جو پروٹوکول کے ذریعے بھیجا جاتا ہے۔ آرٹیفیکٹ بھیجنے کے بعد ریموٹ ایجنٹ کے ساتھ کنکشن بند ہو جاتا ہے جب تک دوبارہ ضرورت نہ ہو۔

#### ایونٹ کیو

یہ جزو **اپ ڈیٹس سنبھالنے اور پیغامات پہنچانے** کے لیے ہے۔ یہ پروڈکشن میں ایجنٹک سسٹمز کے لیے خاص اہمیت رکھتا ہے تاکہ کام مکمل ہونے سے پہلے ایجنٹس کے درمیان کنکشن بند نہ ہو، خاص طور پر جب کام مکمل ہونے میں طویل وقت لگ سکتا ہے۔

### A2A کے فوائد

• **بہتر تعاون**: یہ مختلف وینڈرز اور پلیٹ فارمز کے ایجنٹس کو ایک دوسرے کے ساتھ رابطہ کرنے، کانٹیکسٹ شیئر کرنے، اور مل کر کام کرنے کی اجازت دیتا ہے، جو روایتی طور پر الگ الگ سسٹمز کے درمیان خود کاری کو آسان بناتا ہے۔

• **ماڈل سلیکشن کی لچک**: ہر A2A ایجنٹ اپنی درخواستوں کے لیے الگ LLM منتخب کر سکتا ہے، جس سے ہر ایجنٹ کو بہتر یا فن ٹونڈ ماڈلز کے استعمال کی سہولت ملتی ہے، برخلاف MCP کے بعض معاملات میں واحد LLM کنکشن کے۔

• **ان بلٹ آتھنٹیکیشن**: A2A پروٹوکول میں آتھنٹیکیشن براہِ راست شامل ہے، جو ایجنٹ بات چیت کے لیے مضبوط حفاظتی فریم ورک فراہم کرتا ہے۔

### A2A کی مثال

![A2A Diagram](../../../translated_images/ur/A2A-Diagram.8666928d648acc26.webp)

آئیں اپنے سفر کی بکنگ کے منظر نامے کو A2A کے ساتھ دیکھتے ہیں۔

1. **صارف کی درخواست ملٹی-ایجنٹ کو**: صارف "Travel Agent" A2A کلائنٹ/ایجنٹ کے ساتھ بات چیت کرتا ہے، مثلاً کہتا ہے، "براہ کرم اگلے ہفتے کے لیے ہونولولو کا پورا سفر بک کریں، جس میں فلائٹس، ہوٹل، اور کرائے کی گاڑی شامل ہو۔"

2. **Travel Agent کی ترتیب**: Travel Agent اس پیچیدہ درخواست کو وصول کرتا ہے۔ یہ اپنے LLM کا استعمال کرتے ہوئے مسئلے کا تجزیہ کرتا ہے اور طے کرتا ہے کہ اسے دوسرے خاص ایجنٹس سے بات کرنی ہوگی۔

3. **ایجنٹوں کے درمیان تعلق**: Travel Agent پھر A2A پروٹوکول کے ذریعے نیچے کے ایجنٹس سے کنکشن کرتا ہے، جیسے "Airline Agent"، "Hotel Agent"، اور "Car Rental Agent" جو مختلف کمپنیز کے بانی ہیں۔

4. **ذیلی کاموں کی تفویض**: Travel Agent ان خاص ایجنٹس کو مخصوص کام بھیجتا ہے (مثلاً "ہونولولو کے لیے فلائٹس تلاش کریں"، "ہوٹل بک کریں"، "گاڑی کرائے پر لیں")۔ ہر ایجنٹ اپنا LLM اور ٹولز (جو خود MCP سرور ہو سکتے ہیں) استعمال کرتے ہوئے اپنا حصہ مکمل کرتا ہے۔

5. **مکمل جواب**: جب تمام نیچے کے ایجنٹس اپنے کام مکمل کر لیتے ہیں، تو Travel Agent تمام نتائج (فلائٹ کی تفصیلات، ہوٹل کی تصدیق، کرائے کی گاڑی کی بکنگ) کو یکجا کرکے صارف کو چیٹ انداز میں مکمل جواب بھیجتا ہے۔

## نیچرل لینگویج ویب (NLWeb)

ویب سائٹس طویل عرصے سے صارفین کے لیے انٹرنیٹ پر معلومات اور ڈیٹا تک رسائی کا بنیادی ذریعہ رہی ہیں۔

آئیں NLWeb کے مختلف اجزاء، اس کے فوائد اور ہمارے سفر کی درخواست کی مثال کے ذریعے دیکھیں کہ NLWeb کیسے کام کرتا ہے۔

### NLWeb کے اجزاء

- **NLWeb ایپلیکیشن (کور سروس کوڈ)**: وہ نظام جو قدرتی زبان کے سوالات کو پراسیس کرتا ہے۔ یہ پلیٹ فارم کے مختلف حصوں کو جوڑ کر جوابات تخلیق کرتا ہے۔ آپ اسے ویب سائٹ کی قدرتی زبان کی خصوصیات کو چلانے والا **انجن** سمجھ سکتے ہیں۔

- **NLWeb پروٹوکول**: یہ ویب سائٹ کے ساتھ قدرتی زبان میں بات چیت کے لیے **بنیادی قواعد کا مجموعہ** ہے۔ یہ جوابات JSON format (اکثر Schema.org استعمال کرتے ہوئے) میں واپس بھیجتا ہے۔ اس کا مقصد "AI ویب" کے لیے ایک سادہ بنیاد بنانا ہے، جس طرح HTML نے آن لائن دستاویزات شیئر کرنا ممکن بنایا۔

- **MCP سرور (ماڈل کانٹیکسٹ پروٹوکول اینڈ پوائنٹ)**: ہر NLWeb سیٹ اپ بھی ایک **MCP سرور** کے طور پر کام کرتا ہے۔ اس کا مطلب ہے کہ یہ دوسرے AI سسٹمز کے ساتھ **ٹولز (جیسے "ask" میتھڈ) اور ڈیٹا** شیئر کر سکتا ہے۔ حقیقت میں، یہ ویب سائٹ کے مواد اور صلاحیتوں کو AI ایجنٹس کے لیے قابل استعمال بناتا ہے، اور سائٹ کو وسیع "ایجنٹ ایکوسسٹم" کا حصہ بنا دیتا ہے۔

- **ایمبیڈنگ ماڈلز**: یہ ماڈلز ویب مواد کو عددی نمائندگیوں میں تبدیل کرتے ہیں جنہیں ویکٹرز (embeddings) کہا جاتا ہے۔ یہ ویکٹرز معنی کو اس انداز میں پکڑتے ہیں جو کمپیوٹرز کے لیے موازنہ اور تلاش ممکن بناتے ہیں۔ یہ خاص ڈیٹا بیس میں محفوظ کیے جاتے ہیں، اور صارفین انتخاب کر سکتے ہیں کہ کون سا ایمبیڈنگ ماڈل استعمال کریں۔

- **ویکٹر ڈیٹا بیس (ریٹریول میکانزم)**: یہ ڈیٹا بیس **ویب سائٹ کے مواد کے ایمبیڈنگز ذخیرہ کرتا ہے**۔ جب کوئی سوال پوچھتا ہے، NLWeb تیزی سے متعلقہ معلومات تلاش کرنے کے لیے ویکٹر ڈیٹا بیس چیک کرتا ہے۔ یہ ممکنہ جوابات کی ایک تیز فہرست دیتا ہے، مماثلت کے حساب سے درجہ بندی کی ہوئی۔ NLWeb مختلف ویکٹر اسٹوریج سسٹمز جیسے Qdrant, Snowflake, Milvus, Azure AI Search, اور Elasticsearch کے ساتھ کام کرتا ہے۔

### NLWeb کی مثال

![NLWeb](../../../translated_images/ur/nlweb-diagram.c1e2390b310e5fe4.webp)

ہمارے سفر کی بکنگ ویب سائٹ کو دوبارہ دیکھیں، لیکن اس بار یہ NLWeb سے چل رہی ہے۔

1. **ڈیٹا امیجیشن**: ویب سائٹ کے موجودہ پراڈکٹ کیٹلاگز (جیسے فلائٹس کی فہرست، ہوٹل کی تفصیلات، ٹور پیکجز) Schema.org فارمیٹ میں یا RSS فیڈز کے ذریعے لوڈ ہوتے ہیں۔ NLWeb کے ٹولز اس منظم ڈیٹا کو لے کر ایمبیڈنگز بناتے ہیں اور انہیں لوکل یا ریموٹ ویکٹر ڈیٹا بیس میں محفوظ کرتے ہیں۔

2. **قدرتی زبان میں سوال (انسانی)**: صارف ویب سائٹ پر آتا ہے اور مینو نیویگیٹ کرنے کے بجائے چیٹ انٹرفیس میں ٹائپ کرتا ہے: "مجھے اگلے ہفتے ہونولولو میں پول کے ساتھ فیملی فرینڈلی ہوٹل تلاش کریں"۔

3. **NLWeb کی پروسیسنگ**: NLWeb ایپلیکیشن یہ سوال وصول کرتی ہے۔ یہ اسے LLM کو سمجھنے کے لیے بھیجتی ہے اور ساتھ ہی ویکٹر ڈیٹا بیس میں متعلقہ ہوٹل کی لسٹنگ تلاش کرتی ہے۔

4. **صحیح نتائج**: LLM ڈیٹا بیس کے تلاش کے نتائج کو سمجھنے میں مدد دیتا ہے، "فیملی فرینڈلی"، "پول"، اور "ہونولولو" کی بنیاد پر بہترین میچز تلاش کرتا ہے، اور پھر قدرتی زبان میں جواب تیار کرتا ہے۔ اہم بات یہ ہے کہ جواب ویب سائٹ کے اصلی ہوٹل کیٹلاگ پر مبنی ہوتا ہے، جعلی معلومات سے گریز کیا جاتا ہے۔

5. **AI ایجنٹ کے ساتھ تعامل**: چونکہ NLWeb ایک MCP سرور کے طور پر کام کرتا ہے، اس لیے کوئی بیرونی AI ٹریول ایجنٹ بھی اس ویب سائٹ کے NLWeb انسٹانس سے جڑ سکتا ہے۔ AI ایجنٹ `ask` MCP میتھڈ استعمال کر کے براہ راست ویب سائٹ سے سوال کر سکتا ہے: `ask("کیا ہونولولو علاقے میں ہوٹل کی طرف سے تجویز کردہ ویگن فرینڈلی ریسٹورنٹ موجود ہیں؟")`۔ NLWeb انسٹانس اسے پراسیس کرے گا، اپنے ریسٹورنٹ کی معلومات کی ڈیٹا بیس کو استعمال کرتے ہوئے (اگر لوڈ کی گئی ہو)، اور ایک منظم JSON جواب واپس کرے گا۔
